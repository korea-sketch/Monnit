2G 또는 3G 게이트웨이 공장 초기화

셀룰러

경우에 따라 Monnit 셀룰러 2G 또는 3G 게이트웨이를 재설정해야 할 수 있습니다. 이 전체 재설정은 APN 프로비저닝을 포함한 모든 구성을 지웁니다. 셀룰러 통신사 네트워크에 연결할 수 없는 게이트웨이를 문제 해결할 때 필요할 수 있습니다.

단계

게이트웨이 전원을 끈 상태(플러그 분리/스위치 끔)로 시작합니다.

  1. 게이트웨이가 꺼진 상태에서 Reset 버튼을 누르고 유지합니다(계속 누름).
  2. (계속 Reset을 누른 채) 게이트웨이를 켭니다.
  3. LED 순서가 표시되도록 합니다: AMBER(황색) LED 3개 > GREEN LED 3개 > 그다음 RED LED 3개가 계속 켜진 상태로 유지
  4. Reset 버튼을 뗍니다.
  5. 버튼을 떼면 3개 LED가 GREEN으로 깜빡여 떼었음을 확인한 후 RED LED 3개로 돌아갑니다.
  6. 모든 LED가 solid RED에서 solid GREEN으로, 마지막으로 flashing GREEN이 될 때까지 Reset 버튼을 누르고 유지합니다.
  7. Reset 버튼을 뗍니다.
  8. 장치가 재부팅되기를 기다립니다.

게이트웨이 구성: 이후 올바른 APN 구성으로 게이트웨이를 프로비저닝해야 할 수 있습니다.

셀룰러 게이트웨이 신호 레벨 값

셀룰러

iMonnit에서 셀룰러 게이트웨이에 사용되는 신호 표시기는 ASU(Arbitrary Signal Unit) 신호 표시기를 사용합니다. 이는 신호 강도를 나타내는 "막대(bars)" 아이콘으로 변환됩니다.

그래픽 표현: iMonnit 온라인 포털에서 ASU 신호 표시기를 5막대 신호 그래픽으로 변환하는 척도(0막대 = 낮음/무신호, 5 = 최고 신호):

  • 5막대: 24 - 31
  • 4막대: 20 - 23
  • 3막대: 15 - 19
  • 2막대: 11 - 14
  • 1막대: 6 - 10
  • 0막대: 0 - 5

RSSI 참조 기술 변환:

  • GSM 네트워크에서 ASU는 RSSI(수신 신호 강도 표시기)에 매핑: dBm = 2 × ASU - 113, ASU 범위 0.31 및 99(알 수 없음/감지 불가)
  • UMTS 네트워크에서 ASU는 RSCP 레벨에 매핑: dBm = ASU - 115, ASU 범위 0.90 및 255
  • LTE 네트워크에서 ASU는 RSRP에 매핑. 유효 ASU 범위 0~97. 1~96 범위에서 (ASU - 143) < dBm = (ASU - 140)

레거시 4G LTE 게이트웨이 연결 안 됨 (고객 자체 데이터 요금제)

셀룰러

게이트웨이에 수리가 필요한 경우, Monnit으로 반품하기 전에 Monnit 지원팀이 지시한 최선의 문제 해결 단계를 거쳐야 RMA가 발급됩니다.

4G LTE 게이트웨이가 iMonnit Online에 연결되지 않는 경우

4G LTE 게이트웨이는 Cat M-1 또는 NB-1 프로토콜로 작동하는 지원 통신사 네트워크에 연결됩니다. 이 프로토콜을 사용하는 일부 통신사는 게이트웨이가 자동으로 받는 프로비저닝(*Wildcard APN 구성)을 제공하므로 수동 프로비저닝이 필요 없습니다. 활성 SIM이 있는 게이트웨이는 지원 LTE 타워 범위 내에 있으면 자동으로 통신사 네트워크에 연결되어 UDP 포트 3000을 통해 iMonnit Online으로 데이터를 전송합니다.

LTE 게이트웨이와 작동이 확인된 통신사: Verizon, AT&T, Telenor, Telstra, Saskatel, Hologram. Monnit은 추가 통신사를 테스트하지 않아 다른 통신사의 작동은 보장할 수 없습니다. (레거시 LTE 게이트웨이 데이터시트)

단계 요약: iMonnit 계정에 게이트웨이 추가 확인 → 지원 통신사의 활성 LTE 구독 확인 → 전원 연결·시동 테스트 완료 확인 → 지원 LTE 네트워크 감지·연결 확인(solid green Cellular Link LED) → iMonnit 서버로 데이터 전송 확인 → 펌웨어 업데이트.

일부 통신사는 셀룰러 네트워크 통신을 허용하려면 게이트웨이의 IMEI를 SIM 카드에 등록해야 합니다. 미국 Verizon Wireless는 IMEI를 SIM 카드와 활성화해야 합니다. 이를 위해 Monnit이 게이트웨이 IMEI를 Verizon 데이터베이스에 추가해야 하며, support@monnit.com으로 게이트웨이 ID와 함께 요청 이메일을 보내야 합니다.

시동 테스트: 게이트웨이를 켜면 하드웨어 작동을 확인하는 자가 테스트를 수행합니다. LED가 "폭포 디스플레이(waterfall display)"라 불리는 순서를 수행합니다. 완료되면 LTE 네트워크 감지·연결을 시도합니다.

LTE 네트워크 감지·연결 확인 (Cellular Link LED 동작): 자가 테스트 후 하단 LED(Cellular Link)만 켜집니다.

  • Steady Green: 셀룰러 네트워크에 연결됨
  • Triple Blinking Green: 셀룰러 네트워크 오프라인
  • Double Blinking Green: 타워 검색 중
  • Single Blinking Green: 데이터 세션 활성화 중
  • Blinking Green/Red: 낮은 셀룰러 신호
  • Blinking Red: 셀룰러 연결 오류

연결 실패 시: 게이트웨이 전원 끔 → SIM 카드가 단단히 삽입되었는지 확인(인클로저 4개 나사 분리 필요) → Utility 버튼을 15-20초간 눌러 LED가 적색으로 깜빡일 때까지 유지 후 떼기 → 재시작·자가 테스트·LTE 연결 시도. 그래도 안 되면 다른 위치(창가 등)에서 테스트하고, 지속되면 통신사에 문의.

iMonnit 서버로 데이터 전송 확인 (중간 "Server" LED): Steady Green(마지막 통신 OK), Blinking Green(활성 통신), Steady Red(마지막 통신 실패). 하단 LED는 녹색이나 중간 LED가 적색이면 Utility 버튼을 15-20초간 눌렀다 떼기.

펌웨어 업데이트: 초기 펌웨어는 통신사 지원이 제한적이었으므로 가능하면 업데이트하세요. 업데이트하려면 게이트웨이가 iMonnit과 체크인 중이어야 합니다. iMonnit 관리자 로그인 → Gateways → 게이트웨이 선택 → Settings(톱니바퀴) → Commands 하위 탭 → Update 버튼.

셀룰러

참고: 게이트웨이 수리가 필요한 경우 Monnit으로 반품하기 전에 Monnit 지원팀이 지시한 최선의 문제 해결 단계를 거쳐야 RMA가 발급됩니다.

4G LTE 게이트웨이가 iMonnit Online에 연결되지 않는 경우

Monnit은 Verizon(미국), AT&T, AT&T International을 통해 4G LTE 셀룰러 요금제를 제공합니다. 게이트웨이는 활성화되어 셀룰러 네트워크에 연결할 준비가 된 상태로 수령됩니다. 일반적으로 LTE 네트워크 범위 내에서 켜는 즉시 연결됩니다.

4G LTE 게이트웨이는 Cat M-1 또는 NB-1 프로토콜로 작동하는 지원 통신사 네트워크에 연결됩니다. 이 프로토콜은 게이트웨이가 자동으로 받는 프로비저닝을 제공하므로 수동 프로비저닝이 필요 없습니다.

단계 요약: iMonnit 계정에 게이트웨이 추가 확인 → 활성 LTE 구독 확인 → 전원 연결·시동 테스트 완료 확인 → 지원 LTE 네트워크 감지·연결 확인 → iMonnit 서버로 데이터 전송 확인 → 펌웨어 업데이트.

활성 LTE 구독 확인: Monnit에서 셀룰러 서비스를 Monnit에 지불하고 4G LTE 게이트웨이를 구매하면 SIM이 일반적으로 2년 서비스 계약으로 활성화됩니다. 따라서 데이터 요금제 없이 구매한 경우가 아니면 이미 셀룰러 서비스가 있어야 합니다.

(시동 테스트, LED 동작, 데이터 전송 확인, 펌웨어 업데이트 절차는 위 "고객 자체 데이터 요금제" 문서와 동일합니다. 연결 실패 시 동일한 단계 수행 후 지속되면 Monnit 지원팀에 문의하세요.)

Monnit 3G 게이트웨이를 커스텀 APN으로 프로비저닝

셀룰러

Monnit이 제공하지 않는 통신사 요금제로 Monnit 3G 게이트웨이를 사용하려면 게이트웨이가 작동하는 무선 대역에서 M2M 통신을 지원하는 활성 데이터 요금제 SIM이 필요합니다.

  1. iMonnit 계정을 설정하고 셀룰러 게이트웨이 및 무선 센서를 추가합니다.
  2. 선호하는 셀룰러 네트워크에서 SIM 카드를 구하여 활성화합니다.
  3. 4개 나사를 풀어 게이트웨이를 열고 상단을 제거합니다.
  4. SIM 카드를 SIM 홀더에 삽입합니다(금색 접점이 아래로 향하게).
  5. 게이트웨이를 켜기 전에 아래 지침에 따라 셀룰러 게이트웨이를 구성합니다.

Monnit 3G 게이트웨이 지원 대역: 북미 게이트웨이 (B5, B2, B4) = 800/850, 1700, 1900MHz / 국제 게이트웨이 (B5, B8, B1) = 800/850, 900, 2100MHz

셀룰러 게이트웨이 구성: SIM 카드를 넣은 후 휴대전화로 아래 매개변수로 문자 메시지를 작성하여 장치의 휴대전화 번호로 보냅니다. 문자 형식은 [매개변수 이름]:[네트워크별 데이터]; 형태여야 합니다(이름 뒤 콜론, 데이터 뒤 세미콜론). 하나의 문자에 하나 또는 모든 매개변수를 포함할 수 있습니다. 변경이 필요한 매개변수만 보내는 것을 권장합니다. 공백·줄바꿈은 무시되며 매개변수는 대소문자를 구분하지 않습니다.

예: Telefonica 네트워크에 APN만 제공하면 되는 경우 문자는 Apn:m2mkit.telefonica.com; 만 포함.

APN 구성 매개변수:

| 필드 이름 | 설명 | 매개변수 형식 | 예시 | | --- | --- | --- | --- | | Cellular Access Point Name | 제공자가 할당한 M2M 네트워크 이름 | Apn: name here; | Apn:m2mkit.telefonica.com; | | Username | 일부 네트워크에서 필요 | User: username; | User:datauser; | | Password | 일부 네트워크에서 필요 | Pass: password; | Pass: 1234; | | DNS setting | DNS 할당 | Dns: dns; | Dns: 8.8.8.8; | | Poll interval | Monnit 명령 폴링 | Poll: pollrate; | Poll:30; | | Host address | 대상 서버, 기본 sensorsgateway.com | Host: address; | Host:sensorsgateway.com; | | Host port | 데이터 전달 포트, 기본 3000 | Port: port; | Port: 3000; |

문자 작성 후 SIM 카드 전화번호로 전송 → 셀룰러 게이트웨이 안테나 부착 → 게이트웨이 전원 연결. 게이트웨이가 문자로 구성 정보를 받아 메모리에 저장하고, 성공하면 연결되면서 LED가 GREEN으로 바뀌며 구성에 사용된 전화로 응답 문자를 보냅니다. 실패하면 LED가 녹색이 되지 않으며, 게이트웨이 플러그를 빼고 새 구성 문자를 보낸 후 다시 연결하세요.

GSM 국제 셀룰러 게이트웨이 SIM 카드 구성

셀룰러

셀룰러 게이트웨이(CGW)를 SIM에 맞게 구성하는 것은 간단합니다. 열린 CGW에서 사용 중인 SIM으로 SMS를 보냅니다. 아래는 유럽 시장용 지침입니다.

  • 선택한 이동통신 사업자(SIM 제공자)에게 GPRS 네트워크의 APN 이름을 확인합니다(보통: ApnName, userName, password).
  • 새 SIM을 Monnit CGW에 삽입합니다(이 과정 중 CGW 전원은 꺼둠).
  • 아무 휴대전화에서 할당된 SIM 번호로 다음 구문의 SMS를 보냅니다:
  • `apn_config ApnName,User,Password,`
  • (마지막 쉼표를 잊지 말고, apn_config 뒤에 공백은 하나만)

예 (Vodafone GPRS 구성 - ApnName: internet.vodafone.net, User: internet, Password: internet): `apn_config internet.vodafone.net,internet,internet,`

사용자 이름이나 비밀번호가 없으면 "null"로 대체: `apn_config internet.vodafone.net, null, null,`

  • CGW 전원을 켭니다(참고: 처음에는 CGW가 GPRS 연결에 다소 시간이 더 걸립니다).

셀룰러 게이트웨이의 Force High Power 및 Force Low Power

셀룰러

Monnit 셀룰러 게이트웨이는 게이트웨이가 셀룰러 네트워크 연결을 유지하는 방식을 결정하는 Power Mode 기능을 제공합니다. 배터리 전력(해당하는 경우)과 데이터 사용을 효율적으로 사용하기 위한 여러 Power Mode 구성이 있습니다.

Power Mode 구성: iMonnit Online Portal 로그인 → Gateways → 게이트웨이 카드 선택 → Settings → General 탭의 "Gateway Power Mode" → 원하는 모드 선택(Standard/Force Low Power/Force High Power) → Save.

Power Mode 종류:

  • Force High Power: 라인 전원 및 배터리 전원 모두에서 게이트웨이 LED가 켜진 상태 유지. 데이터 전송 완료 후에도 셀룰러 세션 유지.
  • Force Low Power: 라인 전원 및 배터리 전원 모두에서 LED 꺼짐. 각 전송 전에 셀룰러 세션을 열고 전송 완료 후 닫음.
  • Standard: 라인 전원에서는 LED 켜짐·세션 유지. 배터리 전원에서는 LED 꺼짐·각 전송마다 세션 열고 닫음.

반복되는 암호화 구성 메시지 (Repeating Encryption Configuration Messages)

셀룰러

이 반복되는 Gateway Management 메시지의 가장 흔한 원인은 낮은 CR2032 시계 배터리입니다.

일부 2G 또는 3G 게이트웨이는 게이트웨이 메인 보드 밑면에 3.6V CR2032 코인셀 배터리가 있어 정확한 시간 클록과 셀룰러 타워 연결을 유지합니다. 이 배터리를 교체해야 하면 게이트웨이가 셀룰러 타워와 세션을 시작할 수 없습니다.

  1. 게이트웨이 케이스를 엽니다(밑면 모서리 4개 나사 제거).
  2. 메인 보드 밑면에 코인셀 배터리 케이지가 있는지 확인합니다. 있으면 클립 등으로 배터리를 케이지에서 밀어냅니다.
  3. 확인된 양호한 3.6V CR2032 배터리로 교체합니다.
  4. 전체 게이트웨이 재설정을 완료합니다.
  5. 필요하면 게이트웨이를 (재)프로비저닝합니다.

문제가 지속되면 CR2032 배터리를 제거하고 다시 테스트하세요. 필요하면 게이트웨이 ID와 함께 support@monnit.com에 문의하세요.

  • iMonnit 계정에 장치 추가 확인.
  • 게이트웨이 전원 연결·시동 테스트 완료 확인.
  • 유효한 IP 주소로 로컬 Ethernet 네트워크에서 작동하는지 확인(10/100 Mbps 호환 스위치 필요).
  • 아웃바운드 TCP 포트 3000으로 인터넷 트래픽을 허용하고, 네트워크 DNS 서버가 sensorsgateway.com을 해석할 수 있는지 확인.
  • 펌웨어 업데이트.

시동 테스트: 전원을 켜면 "폭포 디스플레이(waterfall display)" LED 순서로 자가 테스트를 수행한 후 DHCP로 IP 주소를 받으려 시도합니다.

유효 IP 확인: 게이트웨이에는 기본 IP 주소가 없으며(Auto IP 기능을 켜지 않는 한) 처음 구성 시 DHCP가 필요합니다. 켜면 하단 LED가 적색 깜빡임 → 녹색 깜빡임 → solid green이 됩니다(유효 IP 획득·통신 가능). 적색이면 IP를 받지 못한 것입니다. 많은 Cisco 라우터/스위치에서 spanning-tree portfast 기능을 활성화하세요.

포트 3000 및 DNS: iMonnit Online과 함께 사용하려면 아웃바운드 TCP 포트 3000으로 트래픽을 전달할 수 있고 sensorsgateway.com을 해석할 수 있어야 합니다. IP 화이트리스트가 필요한 네트워크의 공인 IP: 166.117.35.68, 166.117.206.175.

MODBUS TCP 인터페이스 사용법

Ethernet Gateway

이 문서는 ALTA Ethernet Gateway 4용입니다(Ethernet Gateway 3의 MODBUS TCP 인터페이스와는 약간 차이가 있음).

Monnit ALTA Ethernet Gateway 4는 MODBUS TCP 폴(poll) 요청으로 게이트웨이·센서 정보에 직접 접근하는 MODBUS TCP 인터페이스를 제공합니다. MODBUS TCP는 Ethernet에서 작동하는 TCP 인터페이스를 가진 MODBUS RTU 프로토콜입니다. Default Server Interface(iMonnit 소프트웨어) 없이 독립형으로 사용할 수 있습니다. MODBUS TCP 인터페이스는 읽기 전용이므로 이를 통해 데이터를 보내거나 게이트웨이·센서를 구성할 수 없습니다. iMonnit 없이 사설 네트워크에서 게이트웨이를 운영하려면 게이트웨이를 unlock해야 하며 센서 구성에 제약이 있습니다.

MODBUS TCP 인터페이스 활성화: 기본적으로 비활성. 일관된 IP에서 폴링하려면 고정 IP로 구성해야 합니다.

  • iMonnit 사용: iMonnit 로그인 → Gateways → 게이트웨이 선택 → Settings(톱니바퀴) → Interface 하위 탭 → "Activate MODBUS Interface" On → Save → Continue. (기본값: TCP Timeout 5분, Port 502 — Modbus 하위 탭에서 변경 가능.)
  • HTTP 인터페이스(로컬 구성 페이지) 사용: 브라우저에서 게이트웨이 IP 접속 → Data Interfaces 탭 → MODBUS TCP Interface 탭 → Active 선택 → 구성 입력(Data Expiration: 레지스터 데이터 유효 분, Socket Timeout: 서버 통신 대기 초, Listening Port) → Save Changes.

Auto Reset: 기본적으로 Ethernet 게이트웨이는 168시간(7일) Auto Reset 주기를 가져 주 1회 재설정되며, 이때 모든 MODBUS TCP 데이터 레지스터가 null로 지워집니다. 주 1회 데이터가 null로 지워지면 Local HTTP 구성 페이지에서 Auto Reset을 0으로 설정해 비활성화하세요.

동시 연결 1개: MODBUS TCP 인터페이스는 한 번에 하나의 TCP 연결만 지원하므로 한 IP 주소에서만 폴링할 수 있습니다.

MODBUS 레지스터: 모든 데이터 값을 16비트 레지스터에 저장합니다. 16비트보다 큰 값(예: 32비트 장치 ID)은 여러 레지스터에 분산되어 연결(concatenate)해야 합니다. Monnit ALTA 게이트웨이는 MODBUS 서버입니다.

게이트웨이 레지스터 (40001~40005): Gateway ID_High(40001), Gateway ID_Low(40002), Version Revision+Major(40003), Version Minor+Release(40004), Device Count(40005).

센서 레지스터: 각 센서마다 16개 레지스터 뱅크를 차지하며, 슬롯 1 센서는 40101~40116을 사용합니다. 8개는 센서 정보, 8개는 판독 데이터용입니다. 주요 필드: Sensor ID_High(40101), Sensor ID_Low(40102), Device Type(40103), Data Age(40104, 마지막 데이터 이후 경과 초), Device Active(40105), Is Aware(40106), Voltage(40107, 센티볼트), RSSI(40108, 신호 강도 %), Data 1~8(40109~40116). 슬롯 2 센서는 40117~40132, 슬롯 3은 40133~40148 식으로 이어집니다.

RSSI 해석: -115 이하 = 0%, -111 = 10%, -107 = 20%, -103 = 30%, -99 = 40%, -94 = 50%, -90 = 60%, -80 = 70%, -70 = 80%, -60 = 90%, -50 이상 = 100%. (펌웨어 1.0.6.5 이전은 역백분율로 보고.)

분할 레지스터 연결 예시: 슬롯 2의 ID 315234 센서 — 40117(=3 십진), 40118(=53090 십진). 16진수로 변환: 3, CF62 → 연결하면 3CF62 → 십진 변환 시 315234.

습도 센서 예시: ID 315234, 슬롯 2, 5분 전 체크인, Not Aware, 신호 60%, 배터리 90%, 판독 22.5°C·42.5% 습도. 폴링 결과: 40119=48(Humidity Sensor 타입), 40120=300(데이터 300초 경과), 40121=1(활성), 40122=0(Not Aware), 40123=316(3.16V), 40124=60(RSSI -60), 40125=4250(42.5% 습도), 40126=2250(22.5°C), 40127~40132=0(미사용).

인터페이스 테스트: Schneider Electric의 경량 도구로 테스트할 수 있습니다(TCP 포트 502 기본). PC에서 게이트웨이 IP를 ping → Tester.exe 실행 → Port TCP/IP, 게이트웨이 IP 입력, Holding Registers(R03/W16), Starting Register 1, 5개 레지스터 Read → 게이트웨이 정보 반환. 센서 슬롯 1은 Starting Register 101, 10개 레지스터.

펌웨어 업데이트: MODBUS TCP 인터페이스에 중요한 업데이트가 있으므로 센서 네트워크 구축 전 최신 펌웨어 확인이 필수입니다(펌웨어 1.0.5.1 이상은 업데이트 후 구성 복원 가능).

게이트웨이 펌웨어 업데이트는 구성을 기본 설정으로 재설정할 수 있습니다. 고정 IP로 프로그래밍된 게이트웨이는 IP가 지워지고 DHCP가 다시 활성화될 수 있습니다(ALTA Ethernet Gateway 3, 2G/3G 셀룰러 게이트웨이 펌웨어 1.4.3.0까지, EGW4 펌웨어 1.0.5.8 이전에 해당). 커스텀 APN의 3G/2G 게이트웨이는 APN을 다시 보내야 합니다.

  1. iMonnit에 관리자로 로그인.
  2. 왼쪽 사이드바에서 Gateways 클릭. (업데이트 가능하면 Gateways 옆에 다운로드 화살표 아이콘이 표시됨.)
  1. 다운로드 화살표 아이콘 클릭하여 업데이트할 게이트웨이 확인.
  2. 일부만 업데이트하려면 해당 게이트웨이를 클릭한 후 Update Selected 클릭(녹색 체크 표시 = 선택됨).

ALTA Ethernet Gateway 4는 MODBUS TCP와 SNMP 인터페이스를 제공하여 MODBUS TCP/SNMP 네트워크에서 센서 데이터를 폴링할 수 있습니다. 이 인터페이스들은 기본적으로 비활성화되어 있으나 고정 IP 구성 시 iMonnit 소프트웨어나 HTTP 인터페이스로 활성화할 수 있습니다. Default Interface(iMonnit Online)는 항상 활성화되어 있으나, Monnit 소프트웨어를 사용하지 않는 사설 네트워크에서 운영하면 비활성화할 수 있습니다.

Default Server Interface 비활성화: Default Interface 비활성화 전 항상 펌웨어를 업데이트해야 합니다. 게이트웨이가 Monnit 소프트웨어로 데이터를 보내지 않으면서 MODBUS TCP/SNMP 인터페이스에서 센서 데이터를 받고 폴링에 응답하려면 Default Interface를 비활성화해야 합니다. 게이트웨이를 unlock해야 이 옵션이 보입니다. 사설 네트워크에서 이 인터페이스를 활성 상태로 두고 Monnit 서버에 도달할 수 없으면, 게이트웨이가 계속 iMonnit에 도달을 시도하다 센서 네트워크가 비활성화됩니다(센서 데이터 미갱신, 상단 LED 꺼짐). Unlock 후 HTTP 인터페이스 → Data Interfaces 탭 → Default Server Configuration → Off → Save Changes로 비활성화.

게이트웨이가 제공하는 5개 데이터 인터페이스:

  1. Default Server Interface (iMonnit Online/Enterprise/Express/MINE) — TCP 포트 3000으로 iMonnit과 통신.
  2. Modbus TCP — 게이트웨이·센서 데이터를 폴링하는 읽기 전용 인터페이스. 16비트 레지스터에 저장. 고정 IP 필요. Server ID 구성 미지원.
  3. SNMP — SNMP V1 인터페이스로 SNMP 프로토콜로 센서 데이터 폴링. 기본 community string은 "public"(HTTP 인터페이스에서 변경 가능). Walk 명령 지원, MIB 다운로드. SNMP Trap을 특정 엔드포인트로 보낼 수 있음.
  4. SNTP — 네트워크 동기화 컴퓨터 클록. 펌웨어 1.0.5.5 이전 버전은 Default Interface 비활성화 시 SNTP를 구성해야 함(상태가 "On and Synced"여야 센서 네트워크 활성). 1.0.5.5 이상은 Default Interface 비활성화 시 SNTP 불필요.
  5. HTTP Interface (로컬 구성 페이지).

Auto Reset: 기본적으로 활성화되어 7일(168시간) 주기. 주 1회 재설정 시 MODBUS TCP 레지스터의 센서 데이터가 지워짐. HTTP 인터페이스에서 0으로 설정해 비활성화 가능.

센서 구성: Default Server Interface만이 센서 구성 기능을 제공합니다. 따라서 기본 설정에서 센서 구성을 변경하려면 Monnit 소프트웨어가 필요합니다. 사설 네트워크에서(Default Interface 비활성화) 120분보다 빈번한 Heartbeat 등을 원하면, Default Interface 비활성화 전에 Monnit 소프트웨어로 센서를 구성해야 합니다. HTTP 인터페이스로는 센서를 구성할 수 없습니다. Heartbeat 구성은 사용 소프트웨어 버전에 의해 제한됩니다(iMonnit Online 최소 1분(HX Credits) 또는 10분(Premiere), 로컬 버전(Enterprise/Express/MINE) 최소 1초).

Ethernet Gateway 4 보안 브리프

Ethernet Gateway

Monnit ALTA® Ethernet Gateway 4(EGW4) Ethernet 네트워크 인터페이스는 다음으로 구성됩니다: 독점 운영 체제, WIZnet 하드와이어드 TCP/IP 스택, 구성 가능한 소켓 인터페이스, 애플리케이션별 보안. 모두 오늘날의 IoT 보안 요구사항을 충족하도록 선택·설계되었습니다.

독점 운영 체제(OS): EGW4는 독점 OS와 정적 파일 시스템을 가진 목적 제작 임베디드 게이트웨이입니다. 프로그램이나 바이러스를 추가할 수 있는 메커니즘이 없습니다. 잠재적 공격 지점은 장치 구성 변경과 장치 업그레이드 메커니즘을 통한 코드 교체이므로, 게이트웨이 인터페이스가 이런 변경 방식을 보호합니다.

WIZnet 하드와이어드 TCP/IP 스택: 플러딩, 스푸핑, 인젝션 등의 네트워크 공격을 방지하는 칩 수준의 "공격 불가능한" 하드웨어 네트워크 엔진입니다. 하드웨어 TCP/IP Offload Engine(TOE)은 과도한 플러딩 패킷을 폐기하여 IoT 시스템을 보호하며, 최대 8개의 독립 하드웨어 소켓을 동시 지원합니다.

구성 가능한 인터페이스:

  1. System Socket Interfaces: 표준 DHCP, DNS, SNTP 프로토콜. 기본적으로 DHCP·DNS 사용, SNTP 비활성. DHCP·DNS 소켓은 직접 비활성화 불가하나 고정 IP·IP 주소 사용 시 사실상 비활성.
  2. Default Server Interface: 서버 통신용 간헐적 아웃바운드 TCP 포트 1개. 기본 활성, 아웃바운드 TCP 포트 3000. Unlock된 게이트웨이에서 비활성화 가능.
  3. Modbus TCP Interface: 구성 가능한 포트 번호의 TCP 리스닝 소켓 1개. 기본 TCP 포트 502, 기본 비활성.
  4. SNMP v1 Interface: UDP 리스닝 소켓 1개와 최대 3개의 UDP Trap 송신 소켓. 기본 비활성, SNMP 요청 포트 161·Trap 포트 162.
  5. HTTP Interface: 로컬 웹페이지. 기본 TCP 포트 80에서 활성, 기본 "read-only".

애플리케이션별 보안:

  • Default Server Application: EncryptRF©와 독점 Monnit Server(MSVR) 프로토콜로 통신. 게이트웨이 구성·업데이트는 이 보안 링크로만 허용되며 새 코드 무결성 검증.
  • Modbus TCP / SNMP v1: 읽기 전용 접근만 지원(쓰기 불가).
  • HTTP Application 접근 구성: "Read-Only"(GET만 처리, POST 무시), "5-Minute Access", "30-Minute Access"(부팅/Utility 버튼 후 일정 시간 POST 수락), "Unlimited Access"(항상 POST 수락).

기본 활성 인터페이스 요약: DHCP(UDP 67), DNS(UDP 53), Default Server(TCP 아웃바운드 3000, EncryptRf©·MSVR 프로토콜로 보안), HTTP(TCP 인바운드 80, 기본 read-only).

iMonnit에서 별도 센서 네트워크로 센서를 게이트웨이에 지정하기

Ethernet Gateway

센서가 게이트웨이와 통신하도록 승인되는 방식은 센서 네트워크(Sensor Network)를 통해서입니다. iMonnit Online에서 같은 네트워크에 추가된 센서는 그 네트워크의 게이트웨이와 통신하도록 승인됩니다. 센서가 여러 게이트웨이가 있는 같은 네트워크에 추가되면 그 네트워크의 모든 게이트웨이가 센서 통신을 승인하고, 센서가 통신할 게이트웨이를 자동으로 선택합니다. 따라서 한 센서 네트워크에 여러 게이트웨이가 있으면 센서가 통신할 게이트웨이를 지정할 방법이 없습니다. 이것이 센서가 통신할 게이트웨이를 제어하기 위해 별도 센서 네트워크를 만드는 것을 고려하는 이유입니다.

참고: 계정 또는 Location당 네트워크 5개 제한이 있습니다.

Ethernet 게이트웨이 데이터 만료(Data Expiration)

Ethernet Gateway

Ethernet 게이트웨이의 Data Expiration 구성은 데이터가 지워지기 전까지 센서 판독값이 데이터 레지스터에 얼마나 오래 존재할지 결정합니다. 이는 Ethernet 게이트웨이의 MODBUS TCP 및 SNMP 인터페이스에 적용됩니다. 각 센서 판독값은 다음 중 하나가 발생할 때까지 데이터 레지스터에 저장됩니다.

  • 센서가 새 데이터를 보고하여 이전 데이터를 덮어씀
  • 게이트웨이 재부팅(게이트웨이 센서 네트워크의 모든 센서 데이터 레지스터 지워짐)
  • Data Expiration 타이머 만료
참고: 데이터 레지스터가 지워지면 이를 "0" 판독값으로 오해하기 쉽습니다. 예를 들어 온도 센서의 경우 빈 레지스터를 "0" 또는 "0°C/32°F"로 잘못 해석할 수 있습니다.

일반적 사용: 가장 흔한 사용법은 센서 비활성을 나타내는 것입니다. 예를 들어 센서를 60분 Heartbeat로 구성했다면 Data Expiration을 3시간(3 × 센서 Heartbeat)으로 입력합니다. 센서가 3 Heartbeat(3시간) 동안 연결을 잃으면 레지스터가 지워지며, 이때 센서를 조사하여 비활성 원인을 파악합니다. 센서 레지스터의 Data Age와 함께 사용하면 비활성 문제를 명확히 파악할 수 있습니다.

센서 해석 - MODBUS TCP

Ethernet Gateway

게이트웨이의 MODBUS TCP 인터페이스는 Ethernet Gateway 4 장치에서 센서 데이터를 직접 폴링할 수 있습니다. 센서 레지스터에 기록된 데이터는 원시(raw) 데이터이므로 해석이 필요합니다. Monnit은 센서 Application ID별로 정리된 해석 문서(Excel)를 제공합니다. MODBUS TCP 해석 문서 다운로드

문서 탭: Definitions(프로토콜 정의), Registers(게이트웨이가 사용하는 모든 레지스터 정의), Sensor Data Interpretation(지원 센서 유형 목록과 데이터 해석 정보), Bugs(예기치 않은 동작).

Sensor Data Interpretation 열: Profile(센서 유형의 Application ID), Sensor Name, Data Type(정수 유형), Data Points Presented(값에 대한 상세 설명), Unit(측정 단위), Interpretation(원시 데이터 처리 공식 및 참조 Data 레지스터), Concatenation(여러 레지스터에 분산된 값 연결 예시).

Data Type 종류: u8(부호 없는 8비트), s8(부호 있는 8비트), u16, s16, u32, s32, u64.

⚠ 참고: MODBUS TCP 인터페이스 레지스터는 16비트입니다. 따라서 32비트 또는 64비트 값은 여러 16비트 레지스터에 걸쳐 제시되며 연결(concatenation)이 필요합니다. 분산된 값은 "high"와 "low"로 설명되며 최상위 비트는 "_H", 최하위 비트는 "_L" 접미사가 붙습니다(예: 64비트 Pulse Counter). 8비트 값이 한 레지스터의 상위 4비트와 다음 레지스터의 하위 4비트에 분산되는 경우도 있습니다(예: 3상 전류계).

예시 (Resistance 센서, Application ID 70): 저항 값은 Ohms를 나타내는 32비트 숫자로, Data 0과 Data 1 레지스터에 Resistance_L과 Resistance_H로 분산됩니다(Data 2-7 미사용).

  • Data0 = 12766, Data1 = 16 폴링.
  • 16진수 변환: 16 = 0x0010, 12766 = 0x31DE.
  • 연결: [16]+[12766] → [0x0010]+[0x31DE] = 0x001031DE.
  • 10진수 변환: 0x001031DE = 1,061,342.
  • 공식 적용: 1,061,342 / 10 = 106,134.2.
  • 결과: 저항 = 106,134.2 Ω (범위 0 ~ 250,000.0 Ω, 1자리 소수 해상도를 위해 10으로 나눔).

Monnit Ethernet Gateway 4 SNMP 인터페이스 이해

Ethernet Gateway

SNMP(Simple Network Management Protocol)는 네트워크 장치를 효율적으로 관리하는 도구입니다. 이 문서는 게이트웨이의 SNMP v1 설정 조정 과정을 안내합니다. (접근·활성화·테스트 방법은 이 문서 참조.)

  • Inbound IP Range: SNMP 클라이언트와 통신이 허용되는 IP 주소 범위 정의. 단일 장치면 시작·종료 IP가 동일.
  • Inbound Port: 게이트웨이 서버 데이터를 받을 포트 번호 지정.
  • SNMP Community String: 허용된 IP 범위 내 클라이언트의 구성 가능한 비밀번호 역할. community string이 일치해야 통신 허용. 기본값은 "public"이나 보안을 위해 변경 권장.
  • Trap 설정: 인증 실패, 새 센서 데이터, 센서 알람 등의 이벤트 알림 수신. Trap 주소(SNMP 서버 IP)와 Trap 포트 지정.
  • MIB-II System Configuration Strings: 필수는 아니지만 contact, name, location, description 문자열 설정이 식별·정리에 도움.
  • Interface Data Management: 데이터 만료 시간(시간 단위) 설정. 이 기간 경과 후 Modbus·SNMP 데이터가 0으로 재설정.
  • SNTP 설정: Monnit 인터페이스를 사용할 수 없을 때 네트워크 컴퓨터 클록 동기화. SNTP 비활성화 시 iMonnit으로 시간 동기화. SNTP IP 주소(시간을 가져올 서버) 지정.
  • SNMP Client Configuration: monnit.com에서 MONNIT-EGW4.mib 파일을 다운로드하여 EGW4 데이터 포인트와 센서 데이터의 적절한 필드 이름 제공.
  • 데이터 해석: MIB 파일 로드 후 EGW4SensorInfoTranslatedFormat 테이블에서 변환된 정보 폴링, 다른 테이블은 원시 데이터 표시. 원시 데이터 해석은 Monnit에 SNMP Data Interpretation 문서 요청.

  • HTTP 인터페이스 사용: EGW4와 동일한 방식에 셀룰러 통신사 구성이 추가됨.
  • 산업용 IoT 게이트웨이는 Ethernet 포트가 없어 Ethernet 구성을 지원하지 않으므로 일부 지역에서 자동 구성이 안 될 수 있음.

Carrier Preference 수동 구성: iMonnit 로그인 → Gateway > Settings > Cellular > Carrier Preference > Manual.

  • Carrier APN: 통신사가 자동 구성하지 않으면 APN 주소를 요청해야 함. 자동 제공하면 비워두고 SIM Authentication을 None으로, 대역 선택 후 저장.
  • SIM Authentication Type: 기본 None. 통신사가 SIM 인증을 요구하면 프로토콜(PAP/CHAP) 선택 후 사용자 이름·비밀번호 입력.
  • Active Bands: 통신사의 CAT-M1/NB-IoT 대역 선택(올바르지 않으면 연결 안 됨). 모든 대역 선택 시 모든 대역 스캔(완료에 몇 시간 걸릴 수 있음).

IoT Gateway

모듈 재설정은 게이트웨이의 전체 재설정으로, LTE 또는 IoT 게이트웨이가 제대로 보고하지 않을 때 적용됩니다. 셀 타워에 최적으로 도달하도록 게이트웨이를 야외나 창가에 두는 것이 좋습니다.

단계:

  1. 스위치로 게이트웨이 전원 끔.
  2. Utility 버튼을 20초간 누름(전원을 켜는 동안 계속 누름).
  3. 시동 테스트 후 모든 LED가 적색/녹색으로 번갈아 깜빡이면 버튼 떼기.

ALTA 게이트웨이 네트워크 Reform

무선 및 통신

ALTA 게이트웨이의 Reform 명령은 일부 시나리오에서 필요한 기능을 수행합니다.

Download Sensor List - 센서가 게이트웨이와 통신하도록 승인되는 방식: iMonnit에서 센서가 센서 네트워크에 추가된 후, 게이트웨이는 Download Sensor List 기능을 수행해야 합니다. 이는 네트워크의 모든 센서 ID를 다운로드하여 센서가 게이트웨이와 통신하도록 승인합니다. 이 작업은 세 가지 경우에 발생합니다: (1) 게이트웨이를 처음 켤 때, (2) Utility 버튼을 눌렀다 즉시 떼어 수동 체크인할 때, (3) 기본적으로 12시간마다. 센서가 소프트웨어로 추가되고 Download Sensor List 작업으로 게이트웨이 Sensor List에 다운로드되지 않으면 통신 승인을 받지 못합니다. Reform 시 게이트웨이는 ID를 순차적 순서로 다운로드합니다(작은 ID = 낮은 슬롯).

Reform 명령의 두 가지 핵심 기능: (1) 게이트웨이가 Sensor List를 지웁니다(일반 Download Sensor List는 기존 ID를 지우지 않지만, Reform은 첫 작업으로 Sensor List를 완전히 지움). (2) Download Sensor List 명령을 수행합니다.

iMonnit으로 Reform 수행: iMonnit 로그인 → Gateways → 게이트웨이 선택 → Settings 탭 → Commands 하위 탭 → Reform 버튼. (다음 체크인 시 수행됨.)

네트워크 간 게이트웨이·센서 이동: Download Sensor List는 센서 ID를 지우지 않으므로, 게이트웨이나 센서를 새 네트워크로 옮기면 이전 네트워크의 장치들이 범위 내에 있으면 계속 통신을 승인합니다. 따라서 이전 네트워크의 게이트웨이와 새 네트워크로 옮긴 게이트웨이 모두 Reform해야 합니다.

Gen1과 ALTA의 Reform 차이: Gen1 게이트웨이는 라디오 주파수 대역 내 전용 채널에서 작동하므로 Reform 시 가장 깨끗한 채널을 스캔하는 세 번째 기능을 수행합니다(간섭 시 유용). ALTA는 Frequency-Hopping Spread Spectrum으로 여러 주파수에서 통신하므로 이 기능이 필요 없습니다.

Local 구성(HTTP) 인터페이스로 Reform: HTTP 인터페이스 → General Options → Wireless Sensor Network Configurations → Reform Wireless Network.

게이트웨이 Sensor List 이해

무선 및 통신

Sensor List는 센서가 게이트웨이에 데이터를 보내도록 승인받는 방식입니다. 게이트웨이에 저장된 센서 ID 목록으로, 센서의 Join 요청을 게이트웨이가 인식하여 센서가 게이트웨이에 연결되도록 승인합니다.

Monnit 소프트웨어의 Sensor List: 게이트웨이와 센서가 같은 센서 네트워크에 추가되면 게이트웨이가 그 네트워크의 모든 센서 ID를 Sensor List에 다운로드합니다.

게이트웨이가 센서 ID를 다운로드하는 시점: Sensor List는 소프트웨어→게이트웨이 방향으로만 전송됩니다(게이트웨이→소프트웨어 아님). 즉, 게이트웨이 로컬 인터페이스로 센서를 추가하면 ID가 소프트웨어에 동기화되지 않습니다. 다운로드 트리거: 게이트웨이를 처음 켤 때, Run Mode에서 버튼을 눌렀다 즉시 뗄 때, Reform 명령 시, 기본 12시간마다.

Reform 전까지 센서 ID 유지: 게이트웨이는 네트워크가 Reform될 때까지 센서를 무기한 유지합니다. 따라서 장치를 네트워크 간 이동할 때는 이동한 게이트웨이의 네트워크를 Reform해야 합니다.

iMonnit Online 포털의 Sensor List: 게이트웨이의 "Sensors" 탭은 승인된 모든 센서 ID가 아니라 마지막으로 성공적으로 통신한 센서 ID를 포함합니다. 실제 Sensor List를 보려면 게이트웨이 HTTP 인터페이스(Ethernet 게이트웨이만)의 Wireless Sensor Network에 접근해야 합니다.

Ethernet 게이트웨이 Sensor List 형성의 중요 동작 (MODBUS TCP/SNMP 사용자용): (1) 소프트웨어에서 처음 다운로드 시 ID가 숫자 순서로 슬롯에 배열됨(Reform 시 순서가 바뀔 수 있음). (2) 특정 슬롯에 센서를 삽입하면 이후 슬롯의 센서들이 한 칸씩 밀림. (3) 단일 센서를 Sensor List에서 제거할 방법이 없음(특정 슬롯 할당이 필요하면 HTTP 인터페이스로 할당하거나 전체 Reform 후 수동 할당).

ALTA Gateway Management 메시지 정의

무선 및 통신

Monnit 게이트웨이는 게이트웨이 History에 Gateway Management 항목을 기록하는 다양한 작업을 수행합니다. Gateway Management는 각 작업의 식별자입니다.

  • 2 - 센서 ID로 Sensor List 업데이트 (서버에서 센서 ID 다운로드)
  • 3 - Network Configuration 업데이트됨
  • 4 - Reform 네트워크 명령 수신됨
  • 5 - 게이트웨이 공장 기본값 재설정
  • 7 - Wi-Fi 센서 MAC 주소에서 게이트웨이·센서 ID 반환
  • 8 - 게이트웨이 구성에 맞게 데이터베이스 업데이트 (구성 동기화)
  • 10 - 상태 정보 기록 (전력 사용 등)
  • 11 - 게이트웨이 시동 정보 기록
  • 13 - 게이트웨이가 다른 호스트로 포인팅됨 (Enterprise/MINE 등)
  • 14 - 업데이트 다운로드 확인
  • 187 - Gateway GPS Unlock
  • 250 - 게이트웨이 부트로더 시퀀스 (펌웨어 업데이트)
  • 253 - 센서 펌웨어 업데이트/BSN 업데이트
  • 245 - 게이트웨이 셀룰러 모듈 업데이트

유형 식별: 게이트웨이 History 뷰에서 Gateway Management 텍스트 위에 커서를 올리면 메시지 유형 번호가 표시됩니다.

센서 메시지 수: Gateway History의 Messages 열은 해당 체크인에서 게이트웨이가 전달한 센서 통신 수를 나타냅니다(범위 내 센서의 통신 시도 수).

ALTA 게이트웨이 — Trigger on Aware 작동 방식

무선 및 통신

Monnit ALTA 게이트웨이가 센서 판독값을 즉시 iMonnit으로 전송하도록 하는 구성이 Trigger on Aware 설정입니다. 활성화하면 게이트웨이가 Aware State인 센서로부터 판독값을 받는 즉시 Heartbeat를 트리거합니다. 이 기능 덕분에 예약된 Heartbeat를 기다리지 않고 조건 발생 시 알림을 받을 수 있습니다.

끄면 센서가 Aware State 판독값을 게이트웨이에 즉시 보고하더라도 게이트웨이는 예약된 Heartbeat까지 iMonnit 전달을 대기합니다(60분 Heartbeat 셀룰러 게이트웨이는 상당히 오래 기다릴 수 있음).

기본 활성화: 일반적으로 기본 활성화되어 있으나, 아니면: iMonnit 로그인 → Gateways → 게이트웨이 선택 → Settings 탭 → General 하위 탭에서 On Aware Messages를 Trigger Heartbeat로 → Save.

끄고 싶은 경우: 예약된 Heartbeat에만 데이터를 원하거나 셀룰러 데이터 사용을 최소화하려면 Wait for Heartbeat로 토글하세요.

비인가 센서·게이트웨이 통신 (Unauthorized Sensor and Gateway Communication)

무선 및 통신

센서가 같은 네트워크가 아닌 게이트웨이와 통신하는 경우

ALTA 센서는 Sensor List를 통해 ALTA 게이트웨이와 통신하도록 승인됩니다. 게이트웨이가 센서를 Sensor List에 다운로드하면, 네트워크가 Reform될 때까지 센서 ID가 제거되지 않습니다. 따라서 센서를 다른 네트워크로 옮겨도 이전 게이트웨이의 Sensor List에 그 ID가 남아 있어, 범위 내에 있으면 그 게이트웨이와 계속 통신합니다.

게이트웨이 ID 식별: Gateway ID는 Sensor Readings에 기록됩니다. Export 기능으로 센서 데이터를 다운로드하면 Gateway ID 열이 포함됩니다.

Webhook에 미치는 영향: 게이트웨이 Heartbeat가 센서 데이터를 포함한 webhook을 트리거하므로, 같은 네트워크가 아닌 센서가 범위 내에서 게이트웨이와 계속 체크인하면 그 센서 데이터가 webhook JSON에 포함됩니다(예상치 못한 센서 데이터가 보일 수 있음).

Serial MODBUS

Serial MODBUS 게이트웨이는 센서 데이터를 직접 폴링할 수 있으며, 레지스터의 원시 데이터는 해석이 필요합니다. Monnit은 센서 Application ID별 해석 문서(Excel)를 제공합니다. Serial MODBUS 해석 문서 다운로드

문서 탭: Definitions(프로토콜 정의), FRR Decoding Table(Fast Read Registers 해석), WD_Data Register Breakdown(지원 센서 유형별 데이터 해석).

Data Type: u8, s8, u16, s16, u32, s32, u64. ⚠ Serial MODBUS 게이트웨이 레지스터는 16비트이므로 32/64비트 값은 여러 레지스터에 분산되어 연결(concatenation)이 필요합니다("_H"/"_L" 접미사).

예시 (Resistance 센서, Application ID 70): 저항 값은 Ohms를 나타내는 32비트로 Data 0(Resistance_L)과 Data 1(Resistance_H)에 분산됩니다.

MODBUS TCP/SNMP 인터페이스 운영 시 고려사항: 게이트웨이 재부팅 시 센서 레지스터 값이 지워지므로, 사설 MODBUS/SNMP 네트워크에서 기본값(주 1회) 재부팅이면 센서 레지스터에 0 값이 보고될 수 있습니다(예: 온도 센서 0을 0°C로 오해). 이 경우 기능 비활성화를 고려하세요(Data Expiration 타임아웃 경과 시에도 동일).

게이트웨이 Heartbeat를 0분으로 설정하여 센서 메시지 즉시 전송

Settings

기본 구성에서 게이트웨이 Heartbeat는 예약된 일정으로 발생합니다. 게이트웨이가 센서 Heartbeat에 센서 통신을 받아 게이트웨이 Heartbeat에 Monnit 소프트웨어로 전달하기 위해 큐에 넣습니다. 이를 변경해 게이트웨이가 받는 즉시 전송하도록 할 수 있습니다.

Heartbeat를 0분으로 설정: 게이트웨이가 판독값을 받는 즉시 Monnit 소프트웨어로 전송합니다(센서 동작·Heartbeat는 변경 안 됨). 기본적으로 Aware State 메시지는 즉시 전송되지만, 이 구성은 모든 센서 판독값을 즉시 전송하게 합니다.

데이터 한도 고려: 셀룰러 게이트웨이의 경우 Monnit 표준 셀룰러 플랜은 월 1MB입니다. 여러 센서로부터 데이터를 받는 게이트웨이에 이 구성을 적용하면 월 1MB 한도를 쉽게 초과할 수 있습니다.

게이트웨이 유형별 기본 Heartbeat: Ethernet 게이트웨이 5분, 4G LTE 셀룰러 15분, 3G 셀룰러 60분, 2G 셀룰러 60분, 무선 센서 어댑터/USB 게이트웨이 5분. (참고: 모든 게이트웨이 유형이 0 Heartbeat를 지원하지는 않음 — UI에서 0 설정이 안 되면 미지원.)

유용한 경우: WebHook 운영 시. WebHook은 게이트웨이 통신으로 트리거되므로 한 WebHook에 여러 센서 판독값 JSON이 포함될 수 있는데, Heartbeat를 0으로 하면 각 센서 체크인마다 WebHook이 트리거됩니다(파싱이 더 쉬움). 단, 0분에서도 한 통신에 여러 판독값이 전송되는 경우가 있을 수 있으므로 파싱 코드에서 대비 필요.

게이트웨이 Heartbeat가 센서보다 덜 빈번한 경우

Settings

장치 Heartbeat 설정 시 그것이 iMonnit 포털 정보 표시에 미치는 영향을 고려해야 합니다. 센서는 센서 Heartbeat에 게이트웨이와 통신하고, 게이트웨이는 게이트웨이 Heartbeat에 iMonnit과 통신합니다.

센서 Heartbeat가 게이트웨이보다 빈번하면, 게이트웨이가 센서 정보를 받아도 게이트웨이 Heartbeat까지 전송하지 않아 메시지가 지연됩니다. 게이트웨이가 전송할 때 큐의 메시지를 모두 보냅니다. 이 메시지는 올바른 시간으로 타임스탬프되지만 iMonnit 전달은 지연됩니다. (Aware State는 "On Aware Messages"가 Trigger Heartbeat면 즉시 전송됨.)

예시 (센서 Heartbeat 10분, 게이트웨이 15분):

  1. 10:00:00 게이트웨이 체크인·iMonnit 전송
  2. 10:01:00 센서가 게이트웨이에 메시지
  3. 게이트웨이가 메모리에 보관
  4. 10:11:00 센서가 게이트웨이에 메시지
  5. 게이트웨이가 메모리에 보관
  6. 10:15:00 게이트웨이 체크인·iMonnit 전송
  7. 이전 두 센서 메시지가 10:01:00, 10:11:00 타임스탬프로 업로드됨

최종 수정일: 2023년 10월 5일

무선 USB 게이트웨이, 호스트 컴퓨터 절전 모드 및 전원 끄기

USB

MonnitLink™ 무선 게이트웨이나 USB 센서 어댑터가 PC가 꺼지거나 절전 모드에 들어가 센서 전송을 받을 수 없으면, 센서는 게이트웨이와 연결을 계속 시도합니다(기본 2회). 연결할 수 없으면 "link mode"에 진입해 근처의 다른 게이트웨이를 찾습니다. 게이트웨이를 찾지 못하면 센서는 "power save" 모드에 진입하여 2시간 동안 절전한 후 다시 link mode로 게이트웨이를 찾습니다. 이 과정은 센서가 통신할 게이트웨이를 찾을 때까지 반복됩니다.

참고: 게이트웨이가 꺼져 있는 동안에는 데이터가 수집되지 않고 알림도 전송될 수 없습니다.

ALTA 게이트웨이를 iMonnit Enterprise·Express·MINE 호스팅 커스텀 호스트/IP로 잠금 해제·포인팅

Unlocking and Pointing

ALTA 게이트웨이를 커스텀 호스트/IP로 포인팅하기 전에 게이트웨이를 unlock해야 합니다. Gateway Unlock 코드 구매. Unlock되지 않은 게이트웨이는 unlock 명령을 받기 위해 아웃바운드 포트 3000으로 인터넷을 통해 iMonnit과 통신해야 합니다.

Monnit은 데이터가 Monnit 소프트웨어로 전송되는 인터페이스를 Default Interface라고 부릅니다. 게이트웨이는 Default Interface로 한 번에 하나의 대상에만 데이터를 전송할 수 있습니다.

대안: Default Interface 비활성화를 위한 unlock — 일부 ALTA Ethernet 게이트웨이는 Monnit 서버 없이 사설 네트워크에서 사용하기 위해 unlock됩니다(상세).

단계 요약: Unlock 코드 적용 → SetHost 도구로 포인팅 → iMonnit에 연결 → 게이트웨이가 체크인하여 Unlock/Point 명령 수신 확인 → 새 호스트와 통신 확인 → (필요 시) iMonnit으로 재설정.

Unlock 코드 적용:

  • iMonnit Gateway Unlock 기능 사용: Credits > Gateway Unlock 페이지에서 적용.
  • imonnit.com/SetHost 도구 사용 (iMonnit 로그인 필요, 아니면 404): <https://www.imonnit.com/SetHost> 방문 → 라벨의 Gateway ID·Security Code 입력 → Gateway Server Settings 버튼 → 이메일의 Unlock Code를 Unlock Code 필드에 붙여넣기 → Submit Query.

SetHost 도구로 포인팅: (셀룰러 게이트웨이는 도메인/IP가 인터넷으로 접근 가능한 공인 주소여야 함) Enterprise/Express Host Address 필드에 서버 도메인/IP 입력 → 고정 IP가 필요하면 DHCP 버튼으로 비활성화하고 Static IP 입력(4개 구성 모두 필요) → Save.

iMonnit으로 다시 포인팅 (재설정): (1) iMonnit이 게이트웨이 통신을 다시 받도록 준비 — iMonnit.com/SetHost에서 Gateway ID·Security Code 입력 후 Reset Defaults 클릭. (2) 게이트웨이 서버/호스트 구성 재설정 — 대부분 Reset/Utility 버튼을 15~20초 눌렀다 떼면 됨(셀룰러는 SMS, USB는 Monnit Link Gateway Utility).

USB 게이트웨이/무선 센서 어댑터 unlock: SetHost 도구는 USB 게이트웨이를 지원하지 않으므로 Monnit Support가 수동으로 unlock해야 합니다(support@monnit.com에 Gateway ID·Unlock Code 전송). USB 게이트웨이는 unlock 상태를 unlock한 컴퓨터의 파일에 저장하므로, 다른 컴퓨터로 옮기면 unlock 상태가 무효화됩니다(재요청 필요). Unlock 확인: Monnit Link Gateway Configuration Utility의 Actions 메뉴에 Configure Server Location 옵션이 보이면 성공.

iMonnit에서 게이트웨이 Unlock 코드 적용 방법

Unlocking and Pointing

게이트웨이를 커스텀 호스트/IP로 포인팅하려면(unlock된 상태로 구매하지 않은 경우) 먼저 unlock해야 합니다. Unlock 코드 구매. Unlock되지 않은 게이트웨이는 포트 3000으로 iMonnit과 통신해 unlock 명령을 받아야 합니다.

지원 게이트웨이: ALTA Ethernet Gateway 3, Ethernet Gateway 4, 2G/3G 셀룰러, LTE, IoT 게이트웨이. 미지원: ALTA 무선 센서 어댑터, Serial MODBUS, Edge.

단계:

  1. iMonnit에 관리자로 로그인.
  2. Gateway Unlock Credit 구매: 왼쪽 메뉴 Credits → Gateway Unlock → 하단에서 Monnit Gateway - Unlock 선택 → Quantity 입력 → Checkout. (체크아웃 후 코드가 이메일로 전송됨.)
  3. Credit 사용: Gateway Unlock 페이지에서 이메일의 Unlock Code를 Activate Code 필드에 붙여넣기 → 확인 페이지에서 OK.
  4. 게이트웨이에 적용: Remaining 열에 사용 가능한 Credit 확인 → 잠금 해제할 게이트웨이 옆 Lock 선택(녹색으로 변함) → Save. (다음 Heartbeat에 unlock됨.)
