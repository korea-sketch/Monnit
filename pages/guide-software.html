iMonnit.com 사용자 이름 또는 비밀번호 분실

Account Management

사용자 이름이나 비밀번호를 잊어 iMonnit.com에 로그인할 수 없으면 아래 단계를 따르세요. iMonnit.com 포털은 Monnit.com 판매 웹사이트와 다른 로그인 자격 증명을 가진 별개의 사이트입니다.

단계 요약: 사용자 이름 검색·비밀번호 재설정 → 사용자 계정 잠금 해제 → 활성 iMonnit Premiere 구독 확인.

사용자 이름 검색·비밀번호 재설정: iMonnit.com 로그인 페이지(<https://www.imonnit.com/Account/LogOnOV>)에서:

  1. Forgot? 링크 클릭.
  2. 사용자 Notification 설정에 추가된 이메일 주소 입력 후 Send Email. (이메일이 여러 사용자에 추가되어 있으면 가장 최근 생성된 사용자에 대해 재설정 실행. 여러 사용자가 의심되면 support@monnit.com 문의.)
  3. 이메일로 전달된 Credential Recovery Authorization Key 확인.
  4. 키를 Password Recovery 필드에 입력 후 Set Password.
  5. 사용자 이름이 표시되는 페이지 — 사용자 이름이 이메일과 다를 수 있으니 메모. 새 비밀번호 입력·확인 후 Update Password.
  6. 사용자 이름과 새 비밀번호로 로그인.

사용자 계정이 잠긴 경우: 잘못된 비밀번호로 너무 많이 시도하면 잠깁니다 — 7회 실패 시 5분, 추가 10회 실패 시 65분 잠금(공인 IP 주소에 적용). 시간 경과 후에도 안 되면 support@monnit.com에 문의.

"This User is inactive. Multiple users only available with premium subscription." 오류: 다중 사용자는 유료 iMonnit Premiere 구독 기능입니다. 무료 iMonnit Basic은 단일 사용자만 허용하므로, Premiere가 만료되면 Primary Contact를 제외한 모든 사용자가 비활성화됩니다. 다시 활성화하려면 Premiere를 갱신해야 합니다(Primary Contact가 갱신 가능).

iMonnit에서 사용자를 다른 계정에 연결(Linking)

Account Management

iMonnit Online 포털은 단일 사용자를 여러 계정에 연결할 수 있습니다. 사용자를 다른 계정에 연결하면 그 사용자가 커스텀 권한으로 Linked 계정을 보고, Linked 계정의 Action 수신자로 포함되며, 센서를 보고/구성할 수 있습니다. 단일 사용자 이름을 유지하면서 별도 계정의 장치/정보에 접근하려는 사용자(예: 두 지점에서 일하는 식당 직원)에게 유용합니다. Linked 사용자 계정에 Premiere 구독이 필요합니다.

참고: Linked 사용자는 linked 계정에서 권한이 없으므로 볼 수 있는 네트워크를 제한할 방법이 없습니다. 제한이 필요하면 그 계정에 사용자를 생성해야 합니다.

사용자를 계정에 연결: 접근 권한이 없는 계정의 관리자로 로그인 → Users 탭 → 오른쪽 상단 Link Users 아이콘 → 연결할 사용자 이메일 입력 → Submit.

연결 수락: 연결되는 사용자 계정으로 로그인(Linked 사용자도 자기 계정의 관리자여야 함) → 사용자 메뉴(실루엣 아이콘) → Linked Accounts → Accept Link.

Linked 계정 보기: 사용자 메뉴 → Linked Accounts → 보려는 계정 옆 View Account(눈 아이콘).

Linked 계정 나가기: Linked 계정을 볼 때 왼쪽 사이드바에 Admin 옵션이 나타남 → Admin 선택 → Home 아이콘 → Home → 기본 계정으로 복귀.

사용자 측에서 연결 해제: 사용자 메뉴 → Linked Accounts → Remove.

iMonnit Location과 Network 비교

Account Management

Monnit 센서 시스템을 효과적으로 관리하려면 두 가지 기본 구조인 Location과 Network를 이해해야 합니다. Location은 사람·사용자 권한·알림 정책을 구성하고, Network는 센서·게이트웨이 등 물리적 장치와 통신 경로를 구성합니다.

Location이란? iMonnit 환경의 최상위 조직 구조로 모든 사용자·규칙·보고서·장치 네트워크의 기본 컨테이너입니다. iMonnit에서 Location과 Account는 호환적으로 사용됩니다(새 Location 생성 = 새 Account 생성). 계층 구조(Parent Location + sub-Location)로 구성할 수 있습니다.

Network란? Location 내에 존재하는 게이트웨이·센서의 논리적 그룹입니다. 가장 중요한 기능은 장치 간 통신 승인입니다(센서가 게이트웨이에 데이터를 전송하려면 둘 다 같은 Network에 할당되어야 함). 각 Location은 최대 5개 Network를 지원하며, 더 필요하면 sub-Location을 생성합니다(각 sub-Location도 5개 Network). 각 ALTA 게이트웨이는 최대 100개 센서를 지원하므로 Network당 사실상 100개 센서로 제한됩니다.

핵심 구분 (Location vs Network):

  • 주요 역할: Location = 모든 데이터의 컨테이너(주 조직 계정); Network = 통신 승인을 위한 장치 그룹.
  • 계층: Location = Parent 또는 Sub; Network = 단일 Location 내에 존재.
  • 한도: Location = sub-location 제한 없음; Network = Location당 최대 5개.
  • 소유: Location은 사용자·규칙·보고서·맵 소유(O), Network는 미소유(X). 게이트웨이·센서는 Network가 직접 소유.
  • 구독: Location은 별도 전용 구독 필요; Network는 부모 Location 구독으로 커버.
  • 데이터 분리: Location은 완전한 관리·데이터 격리; Network는 장치 수준 통신 분리만.

아키텍처 선택:

  • 시나리오 A (단일 Location + 다중 Network): 단일 연속 물리 사이트를 결속된 팀이 관리하며 장치 통신 경로만 분리하면 될 때 이상적(예: 층별 Network를 둔 다층 사무실, 생산라인·창고·사무실 Network를 둔 공장, 온도 구역별 냉장 시설). 가장 비용 효율적(구독 1개).
  • 시나리오 B (다중 Location + sub-Location): 절대적 관리 격리가 필요할 때(예: 여러 독립 고객을 관리하는 MSP/리셀러, 지리적으로 분리된 기업 지점). 진정한 멀티테넌시 제공하나 Location당 구독 필요로 비용 증가.

구독 모델: 각 Location과 sub-Location은 자체 iMonnit Premiere 구독이 필요하며, 구독은 공유·풀링 불가. 예: 4개 Location(부모+sub 3개)에 각 25개씩 100개 센서면 "Premiere 100" 1개로는 부족하고 "Premiere 25" 4개가 필요.

사용자 권한: 표준 모델은 엄격한 계층(상향식 가시성) — Parent Location 사용자는 sub-Location을 볼 수 있으나 sub-Location 사용자는 Parent나 형제 Location을 볼 수 없음. 기본적으로 새 표준 사용자는 어떤 네트워크도 볼 수 없어 관리자가 명시적으로 권한 부여해야 함. "Linked User"는 단일 자격 증명으로 완전히 별개의 여러 Location에 접근하는 기능이나, 연결 시 대상 Location에 대한 완전·무제한 접근을 부여하므로 신중히 사용해야 함(네트워크별 제한이 필요하면 대상 Location에 표준 사용자 생성).

중요: 사용자·규칙은 Location 간 이동·공유 불가하므로 초기 아키텍처 선택이 반영구적입니다. 센서는 Location 간 이동은 불가하나 같은 Location 내 Network 간 이동은 가능(양쪽 게이트웨이 Reform 필요).

습도 센서 오프셋과 규칙 조건 트리거 및 차트

Charts

펌웨어 14.x 이상의 ALTA 습도 센서는 센서 메모리에 저장되는 교정(calibration)을 지원합니다. 모든 Generation 1 습도 센서와 펌웨어 14.x 이전 ALTA 습도 센서는 센서 펌웨어에 저장되는 교정을 지원하지 않습니다. 이런 구형 센서에 적용된 교정 오프셋은 소프트웨어에 적용되는 Offset만 지원합니다.

중요: 규칙(Rule)은 iMonnit에 표시되는 오프셋 적용 값이 아니라 센서 레벨에서 보고된 값에 따라 트리거됩니다. 따라서 사용자 관점에서는 iMonnit에 정확한 판독값이 표시되지만, 규칙은 오프셋이 포함되지 않은 값으로 트리거될 수 있습니다. Calibrate 기능으로 입력한 교정/오프셋 값에 따라 규칙을 트리거하려면 펌웨어 14.x 이상의 최신 ALTA 센서가 필요합니다.

확인 방법: iMonnit 로그인 → Sensors → 습도 센서 선택 → Settings 탭(기어 아이콘) → Sensor Info 섹션에서 RadioBand에 ALTA 포함 여부와 펌웨어 버전 확인. "ALTA"가 없으면 Generation 1 센서, 펌웨어 14.x 이하면 교정/오프셋 값으로 규칙이 트리거되지 않습니다.

원시 데이터 확인: 센서 Readings를 CSV로 내보내기(History 탭 → 기간 설정 → 클라우드 다운로드 아이콘) → Raw Data 열은 센서 레벨에서 보고된 값(규칙 트리거 기준), Value/Formatted Value 열은 포털에서 적용된 교정/오프셋 값.

차트: 구형 습도 센서는 차트가 원시 데이터 값 기반이므로, 차트 값(원시 데이터)과 Sensor Readings(소프트웨어 오프셋 적용) 간 불일치가 보일 수 있습니다.

Calibrate 탭 차이: 최신 ALTA 센서(14.x 이상)는 Humidity·Temperature Offset(Monnit 센서 판독값과 기준값의 차이)을 입력하며 다음 체크인에 적용(예: 센서 20°C, 기준 18°C → "-2" 입력 후 Save). 구형(14.x 이전)은 Actual Reading(기준값)을 입력하며 소프트웨어에 즉시 적용.

Monnit Communication Check Utility 사용

Gateways

Communication Check Utility는 PC와 iMonnit 서버 간 연결을 테스트하는 경량 PC 애플리케이션입니다. 게이트웨이가 iMonnit 소프트웨어로 보내는 데이터를 모방하여, 장치가 iMonnit 소프트웨어와 통신할 수 있는 네트워크 경로가 있는지 확인합니다. 로컬 호스팅 버전(Enterprise, Express, MINE)에서는 서버가 게이트웨이 통신을 수락하도록 구성되었는지도 확인합니다.

단계: CommunicationCheckUtility.exe 다운로드 → 게이트웨이를 iMonnit 소프트웨어에 추가(미추가 시) → 유틸리티 실행 → 게이트웨이 ID 입력 → 올바른 Gateway Type 선택 → iMonnit Online 테스트면 "Use Defaults" 체크 유지(로컬 호스팅이면 체크 해제 후 서버 IP/도메인·포트 입력) → Check Communication.

테스트 내용 (3가지): (1) 서버가 게이트웨이 통신을 받을 준비가 되었는지 확인, (2) 유틸리티 실행 네트워크가 서버 DNS를 해석할 수 있는지 확인, (3) 적절한 포트(3000)로 통신이 서버에 도달하는 경로가 있는지 확인.

DNS 해석 테스트: Monnit 게이트웨이는 기본적으로 도메인 이름으로 데이터를 전송하며, 게이트웨이 유형·연식에 따라 5개 도메인 중 하나를 해석합니다: sensorsgateway.com, t1.sensorsgateway.com, t2.sensorsgateway.com, u1.sensorsgateway.com, u2.sensorsgateway.com. (Ethernet·USB 게이트웨이는 TCP, 셀룰러 게이트웨이·MoWi Wi-Fi 센서는 UDP로 전송.)

통신 테스트: DNS 해석 성공 후 지정 게이트웨이 ID의 데이터를 모방한 패킷을 서버로 전송 — "TCP/UDP Connect Success" → "Message Sent" → 서버 수신 시 "Message Received".

Enterprise Wireless Gateway Server: 게이트웨이가 Enterprise 계정에 추가되었고 Enterprise Wireless Gateway Server 서비스가 Windows Services.msc에서 실행 중인지 확인하는 데 유용(이 서비스가 포트 3000으로 게이트웨이 통신을 수락).

참고: 이 유틸리티는 게이트웨이 데이터를 모방하지만 PC의 IP 주소에서 전송됩니다. 따라서 PC에서 테스트 성공이 네트워크가 게이트웨이 통신을 완전히 지원한다는 결론은 아닙니다(일부 네트워크 보안 기능이 PC 테스트는 통과시키되 게이트웨이 데이터는 막을 수 있음).

게이트웨이 상태 색상

Gateways

게이트웨이는 iMonnit 포털 왼쪽에 네 가지 색상(녹색, 노란색, 회색, 빨간색)의 수직 막대를 표시합니다.

  • 녹색: 여러 통신을 놓치지 않았고 트리거된 규칙이 없을 때(1회 통신 누락 시에도 녹색 유지 가능). "0 Minutes ago" 타임스탬프는 지난 1분 내 체크인을 확인.
  • 노란색: 배터리로 작동 중이거나, 하드웨어 문제(단락)를 겪을 때.
  • 회색: 센서를 계산하는 게이트웨이가 여러 Heartbeat를 놓쳤을 때(규칙이 비활성 예약 상태여도). 타임스탬프는 마지막 통신 시각이며, 오른쪽 배터리 아이콘은 현재가 아닌 그 타임스탬프 시점의 배터리 레벨.
  • 빨간색: 가장 최근 판독값이 하나 이상의 규칙을 트리거했을 때(온라인/오프라인/배터리 상태와 무관, 규칙이 비활성 예약 상태여도).

Locations and Networks

iMonnit Online의 Network는 센서와 게이트웨이를 그룹화하여 장치 간 통신을 승인합니다. 게이트웨이와 센서가 같은 Network에 추가되면 게이트웨이가 센서 ID를 다운로드하여 통신을 승인합니다. Network는 장치의 조직 구성 요소이기도 합니다(iMonnit Premiere 회원은 여러 Network 가능). 장치는 한 번에 하나의 Network에만 추가할 수 있습니다.

네트워크 구성 설정:

  • Name: 새 네트워크 식별자.
  • Rules Enabled: !!중요!! 이 박스를 체크하면 네트워크에서 알림이 전송되도록 설정.

  • Install Tech Access Cut off Date: 설치 기간 동안 장치 변경 옵션(기본값은 하루 전 날짜).
  • Device Xml Download: 네트워크 장치 목록 XML 파일 다운로드(iMonnit Enterprise에 장치 목록 업로드에 사용).
  • Add Device to Network: 장치 추가(게이트웨이를 센서보다 먼저 추가).

Location과 Account, 그리고 sub-Location과 sub-account는 호환적으로 사용됩니다. sub-Location은 최상위 Location 아래에 위치한 Location으로 계층 구조를 가능하게 합니다.

Locations 기능 접근에는 Premiere 구독이 필요 없지만, sub-Location을 볼 때 Premiere 기능에 접근하려면 최상위 Location에 Premiere 구독이 필요합니다. 각 sub-location은 Premiere 기능 접근을 위해 자체 Premiere 구독이 필요합니다. 각 Location과 sub-location은 최대 5개 네트워크를 가질 수 있습니다. Premiere 구독은 Location당 연간 센서 수 기준입니다.

Premiere 구독: 각 Location과 sub-location마다 별도입니다(공유 불가). 예: 부모에 Premiere 100이 있어도 sub-location과 공유 불가. 4개 Location(부모+sub 3개)에 각 25개씩 100개 센서면, Premiere 100 하나로는 부족하고 각 Location에 Premiere 25 4개가 필요.

iMonnit Premiere 설정 또는 고급 센서 구성이 보이지 않나요?

Premiere Subscription

연간 iMonnit Premiere 구독이 만료되면 iMonnit Basic으로 되돌아가 Premiere의 모든 고급 설정이 사라집니다. 이 경우 새 연간 라이선스 구매로 갱신해야 합니다. (iMonnit Premiere 구독에만 적용되며, Enterprise·Express·MINE은 별도 관리.)

iMonnit Basic 설정: Interval(활동이 없을 때 센서가 게이트웨이와 통신하는 빈도), Sensor Name(센서 고유 이름).

iMonnit Premiere 고급 설정:

  • Activate When: 어떤 센서 판독값이 Aware State를 트리거하는지 결정.
  • Active Between: 센서가 활동하는 시간대(휴면 중에는 통신 안 함).
  • Assessments per Heartbeat: Heartbeat 사이에 센서가 임계값 대비 측정값을 확인하는 횟수(예: Heartbeat 10분 + Assessments 10 → 1분마다 평가, 250 → 최대 빈도 1초마다 평가).
  • Aware State Buffer: 평가가 임계값에 매우 근접할 때 표준 작동과 Aware State 사이를 오가는 것을 방지.
  • Aware State Heartbeat: Aware State 중 센서가 게이트웨이로 데이터를 전송하는 빈도.
  • Display As: 데이터 유형 레이블 설정(예: 화씨 vs 섭씨).
  • Failed transmissions before link mode: 게이트웨이 응답 없이 Link 모드 진입 전까지 보내는 전송 횟수.
  • Heartbeat Interval: 임계값 밖 활동이 없을 때 센서가 게이트웨이로 데이터를 전송하는 빈도.
  • Hysteresis: 표준 작동과 Aware State 사이를 오가는 것 방지.
  • Maximum/Minimum Threshold: 이 값 초과/미만 평가 시 Aware State 진입.
  • The sensor is on: 센서 활동 시간대.
  • Show Dew Point: 이슬점을 데이터에 포함.
  • Synchronize: 소규모 센서 네트워크에서 센서 통신을 UTC 기준으로 동기화(기본값 Off는 통신을 무작위화하여 견고성 극대화).
  • Time Frame: 평가할 분(minutes).
  • Time to Re-Arm (seconds): 트리거 이벤트 후 재무장(re-arm) 전 대기 시간(초).
  • Use Aware State: 사용자 설정 값 밖의 평가 시 Aware State 진입.
  • Use this sensor through a repeater: 리피터와 함께 사용 시 체크(기본 미체크, 펌웨어 2.0.0.0 이전 센서에만 적용).

iMonnit REST API 센서별 JSON 데이터 예시

REST API and Webhooks

Monnit REST API는 iMonnit Premiere·Enterprise 고객에게만 제공되며, 각 센서 프로파일별 데이터를 제공합니다. SensorDataMessages 메서드로 제공되는 데이터 예시를 센서 유형별(Application ID)로 제공합니다. 아래는 JSON 형식이며, REST API는 XML도 제공합니다.

공통 JSON 필드 구조 (모든 센서 메시지 공통):

  • `DataMessageGUID`: 데이터 메시지 고유 ID
  • `SensorID`: 센서 ID
  • `MessageDate`: 메시지 시각(`/Date(epoch_ms)/` 형식)
  • `State`: 센서 상태 코드
  • `SignalStrength`: 신호 강도(%)
  • `Voltage`: 전압, `Battery`: 배터리(%)
  • `Data`: 원시 데이터(파이프 `|` 또는 쉼표로 다중 값 구분)
  • `DisplayData`: 사람이 읽는 표시 데이터
  • `PlotValue` / `PlotValues`: 차트 값
  • `DataValues`: 데이터 값(파이프 구분)
  • `DataTypes`: 데이터 유형(예: TemperatureData, Percentage, Pressure, Geforce, Speed, Frequency, PPM 등)
  • `PlotLabels`: 차트 레이블
  • `MetNotificationRequirements`: 알림 요건 충족 여부
  • `GatewayID`: 게이트웨이 ID

REST API and Webhooks

경량 IoT 프로토콜인 MQTT(Message Queuing Telemetry Transport)로 데이터를 전송·저장하려면 iMonnit MQTT 웹훅을 사용합니다. iMonnit에서 네트워크·프라이빗 클라우드·지정 MQTT 브로커로 안전하게 데이터를 전송합니다. 지원 브로커: AWS IoT Core, Microsoft Azure IoT Hub, Google Cloud IoT Core 등.

기능: 고유 암호화 검증 지문으로 데이터 인증, 개발자 없이 자동 데이터 수집, 이종 소스 데이터 집계, 자동 백업·전송·재시도·검증.

요구사항: 활성 iMonnit Premiere 구독, iMonnit Online으로 데이터를 전달하는 장치, MQTT 브로커가 구성된 인터넷 접근 가능 엔드포인트, 엔드포인트 보안 자격 증명.

Monnit으로 품질 시스템 검증 요건 충족

Reports

Monnit 원격 모니터링 솔루션은 다양한 목표를 달성할 수 있습니다. 일부 애플리케이션은 제품 데이터시트로 검증할 수 있고, 다른 애플리케이션은 더 구체적인 사용 사례 검증 감사가 필요합니다.

FDA 규제 애플리케이션의 표준 검증 세트는 Design(DQ), Installation(IQ), Operational(OQ), Performance(PQ)입니다. Monnit은 애플리케이션별 감사 문서를 생성하지 않지만, 현장 감사자가 이러한 검증에 사용할 수 있는 견고한 플랫폼을 제공합니다. (FDA Quality System Regulation Process Validation 문서: https://www.fda.gov/media/94074/download)

  • Design Qualification (DQ): 많은 애플리케이션에 즉시 사용 가능한 모니터링 솔루션과 데이터시트 제공(모든 사용 사례에 적용되지는 않음).
  • Installation Qualification (IQ): 제품별 설치 사용자 가이드와 일반 설치 모범 사례 제공(예: 냉동고 리드 배치, 쿨러 외부 배터리 설치, 안테나 방향).
  • Operational Qualification (OQ): "올바로 작동하는가?" — 설치 완료 후 감사 필요(Monnit은 설치 감사 서비스 미제공, 타사 리소스 활용).
  • Performance Qualification (PQ): "실제 조건에서 예상대로 수행하는가?" — 제품 사양은 제공하나 현장 검증은 사이트별로 크게 다르므로 타사 리소스 활용.

알림을 받지 못한 이유 확인

Rules

알림을 받지 못한 이유를 확인하는 체크리스트입니다.

네트워크·사용자 구성: 센서 네트워크에 알림이 활성화되어 있는지, 사용자 Notification Details가 알림을 허용하는지, 사용자가 알림을 opt out했는지(Rules History에 표시; support@monnit.com 문의), SMS/음성(유료) 사용 시 Notification Credits가 있는지, 푸시 알림 사용 시 올바른 앱 설치·활성화 여부.

이벤트별 고려: 센서 데이터 내보내기 시 해당 판독값의 Alert Sent 열이 True인지, Sensor Reading 규칙은 센서가 24시간 이상 트리거 중인지, Advanced 규칙(Aware State 기반)은 센서가 48시간 이상 Aware State인지 확인.

지연 알림 전 두 번째 트리거: 지연이 구성되어 있고 지연 시간 전에 두 번째 센서가 규칙을 트리거하면, 시스템은 첫 트리거를 인지한 것으로 가정하고 두 번째 트리거 알림을 억제합니다.

알림 지연(Delay) 기능

Rules

규칙은 지연 기간 후 알림을 보낼 수 있습니다. 트리거 조건이 일정 시간 지속되어야만 알림이 필요한 경우 유용합니다.

사용 사례:

  • 냉장 모니터링: 냉장고 문 개방이나 성에 제거 주기마다 알림을 받지 않고, 문이 일정 시간 열려 있을 때만 알림. (온도 컴플라이언스 시간 초과 방지.)
  • 계층적 수신자: 한 사용자에게 먼저 알리고 응답이 없으면 지연 후 두 번째 사용자에게 알림(독립적 지연 시간으로 여러 사용자 구성 가능).

단계: iMonnit 로그인 → Rules → 규칙 이름 선택 → Tasks 탭(갈래 화살표 아이콘) → 변경할 Task 유형 선택 → "To:" 필드의 "+" 아이콘 → 지연 알림을 설정할 사용자 선택 → Delay 옵션 오버레이에서 지연 선택 → Set → 사용자 알림 아이콘이 녹색이고 옆에 지연 시간 표시 확인.

작동 방식: 트리거 조건 충족 시 알림 없이 규칙 트리거·지연 타이머 시작. 지연 타이머 종료 전 센서가 정상 범위로 돌아오면 규칙 Disarm/Rearm·알림 미발송. 종료 전 정상 복귀하지 않으면 지연 후 알림 발송.

온도 버퍼: 유리 비드 바이알에 온도 센서 프로브(유선 프로브 필요)를 삽입하면 주변 온도 변화 감지 속도가 느려져 원하는 지연 달성에 유용할 수 있습니다.

두 번째 알림이 안 온 이유: 여러 센서가 같은 알림 규칙을 공유할 때, 시스템은 첫 트리거를 "Active Alert" 이벤트 시작으로 처리합니다. 첫 알림이 대기 중(지연)이거나 미확인 상태에서 두 번째 센서가 트리거하면, 이미 알림 중인 것으로 간주하여 후속 트리거를 첫 알림 아래에 그룹화합니다.

Back Online 및 Inactivity 알림

Rules

Inactivity(비활성):

  • 센서/게이트웨이가 구성된 "Inactivity Time" 내에 통신하지 않으면 규칙 트리거.
  • 그 시간 내에 통신하면 트리거 안 함.
  • 트리거 후 장치가 통신하면(Auto-acknowledge 활성화 시) 규칙이 Disarm/Rearm되고, 다시 "Inactivity Time" 내 통신하지 않을 때까지 재트리거 안 함.
  • 권장 최소 "Inactivity Time": 2 × 센서 Heartbeat + 1 × 게이트웨이 Heartbeat + 1분.

Back Online(복귀):

  • 센서가 "Alert after" 시간 경과 전 계속 보고하면 트리거 안 함.
  • "Alert after" 시간 내 통신하지 않으면 센서의 다음 통신 시 규칙 트리거.
  • Auto-acknowledge 활성화 시, 센서가 "Alert after" 분 전에 두 번째 연속 통신하면 Disarm/Rearm.
  • 권장 최소 "Alert after": 2 × 센서 Heartbeat + 1 × 게이트웨이 Heartbeat + 1분.

둘 다 센서의 이전 Heartbeat에서 타이머가 시작되므로 Inactivity Time과 "Alert after"를 같은 값으로 두는 것이 합리적입니다(차이: Inactivity는 시간 경과 시 즉시 알림, Back Online은 타이머 경과 후 다음 Heartbeat에 알림). 더 복잡한 동작은 System Action으로 구현할 수 있습니다.

Rule Snooze 시간을 0으로 설정

Rules

Snooze 기능은 규칙이 Disarm되지 않으면 설정 간격으로 반복 알림을 보냅니다. 기본 Snooze 시간은 60분입니다.

Snooze를 0으로 설정하면? Snooze를 비활성화하지 않고 기본값 60분으로 설정됩니다. 즉, 0으로 설정하면 조건이 정상 복귀하거나 규칙을 Disarm할 때까지 60분마다 반복 알림을 받습니다.

반복 알림을 원하지 않으면? System Action으로 규칙을 자동 Disarm(Acknowledge)하도록 설정: System Action 활성화 → Action 구성에서 Acknowledge 선택, 원하는 Delay time 선택, 현재 Action to Disarm 선택.

참고: System Action을 활성화하고 싶지 않으면 Snooze 시간을 65535로 설정할 수 있습니다(반복 알림은 여전히 발송되나 45.51일 지연).

Merge Field로 알림에 구체적 세부 정보 포함

Rules

Merge Field(Text Override Values)는 규칙·알림 메시지에 특정 구문의 텍스트를 입력하면 트리거 이벤트의 정보로 대체되는 기능입니다. 수신자에게 트리거 조건별 정보(센서 ID·이름·날짜·판독값 등)를 전달할 수 있습니다.

작동 방식: `{ }` 안에 특정 텍스트로 입력하며, 알림 전송 시 iMonnit이 해당 정보로 대체합니다. 예: Message 필드에 `"Temperature Data Greater Than 32 {ID} {Reading} {Time} {ReadingDate} {ReadingTime} {Name} {Reading}"` 입력. (Rules > 규칙 선택 > Tasks > Edit Email/Edit Text에서 메시지 제목 오른쪽 Merge Fields 버튼으로 확인.)

보안상 HTML 특수 문자는 미지원될 수 있습니다. 예외는 ">"와 "<"(앞뒤에 공백이 있을 때만 알림 텍스트에 포함 가능).

iMonnit에서 규칙 Disarm 및 Rearm 방법

Rules

  • Disarm: 트리거 조건이 지속되어도 구성된 Snooze 시간의 반복 알림 전송을 중지.
  • Rearm: 규칙을 재설정하여 트리거 조건이 다시 감지되면 알림 발송.

일반 에스컬레이션: (1) iMonnit이 판독값을 받아 에스컬레이션 시작 → (2) 수신자에게 구성된 알림 트리거(트리거 조건 감지부터 실제 발송까지 1분 지연) → (3) 알림 트리거 시 왼쪽 상단 "종" 아이콘에 빨간 점, 관련 트리거 장치 상태 점도 빨간색 → (4) 조치 없으면 Snooze 타이머(기본 60분)로 반복 알림 → (5) 종 아이콘 → Disarm 종 아이콘으로 반복 알림 중지 → (6) 센서가 조건 종료 보고 시 Rearm(Auto Acknowledge 활성화 시) → (7) Disarm한 규칙은 수동 Rearm 가능.

규칙 색상 코딩:

  • 노란색: Disarm되었으나 Rearm되지 않고 여전히 트리거 중인 트리거에 연결된 규칙(Snooze 시간에 알림 미발송).
  • 빨간색: Disarm/Rearm되지 않고 여전히 트리거 중인 트리거에 연결된 규칙(Snooze 시간에 알림 계속 발송). 규칙이 비활성·미발송이어도 트리거 센서가 조건 충족 시 빨간색 표시 가능.

24시간 후 중지: 알림이 24시간 내 Disarm/Rearm되지 않으면 발송이 중지됩니다(Monnit은 Snooze 알림 24시간을 조건 대응에 충분한 시간으로 간주). 이후 Rearm하면 다시 트리거되나, Rearm하지 않으면 더 이상 트리거되지 않습니다.

Advanced Rule(고급 규칙) 유형 이해

Rules

Advanced Rule은 센서의 Aware State(및 기타 기록된 조건)로 트리거되며, 일반적으로 센서의 Aware State Threshold와 직접 연결됩니다. 더 복잡한 로직을 위한 여러 유형이 있습니다.

  • Notify After Aware Period (가장 일반적): 센서의 Aware State로 트리거(특정 판독값 아님). 타이머는 첫 Aware 메시지로 시작하고, 타이머 경과 후 후속 연속 Aware 메시지로 트리거됨.
  • Back Online: 센서가 사전 정의된 시간 동안 오프라인이었다가 다시 온라인되면 트리거(예: 게이트웨이 범위 밖으로 이동 후 연결 복구 시).
  • Gateway On Battery: 게이트웨이가 라인 전원을 잃고 배터리 백업으로 전환되면 트리거(배터리 백업 게이트웨이 필요).
  • Frequent Aware Messages: 사전 구성된 시간 동안 사전 구성된 수의 Aware 메시지가 전송되면 트리거.
  • First Aware Message: 장치가 처음 Aware State에 진입(Aware 메시지 전송)할 때 트리거. 센서가 Aware State 밖으로 보고할 때까지 후속 Aware 메시지는 미트리거.
  • First Non-Aware Message: 장치가 Aware State에서 처음 나올 때 트리거.
  • Aware State Changed: Aware State 변경(진입 시 첫 Aware 메시지, 진출 시 첫 비-Aware 메시지)으로 트리거.
  • Gateway Switched to Line Power: 게이트웨이가 라인 전원에 연결되어 배터리 백업을 더 이상 사용하지 않을 때 트리거(배터리 백업 게이트웨이 필요).
  • Notify After Not Aware Period: 타이머는 Aware State 밖의 첫 메시지로 시작, 경과 후 후속 연속 비-Aware 메시지로 트리거.
  • Assigned Sensors Meet Aware Threshold: 할당된 센서 중 선택 상태인 센서 비율이 지정 비율 초과/미만(구성에 따라)이면 트리거.
  • Aware Change Count: Aware State 변경 횟수가 구성된 임계값을 초과하면 트리거(이후 카운트 재설정).

비활성(Inactivity) 알림 시간 설정

Rules

Inactivity 시간은 센서가 실제로 비활성이 아닐 때 알림받는 것을 방지하는 중요한 고려사항입니다. iMonnit 소프트웨어가 카운트하며, 통신 체인(센서 Heartbeat > 게이트웨이 Heartbeat > iMonnit) 지연 때문에 센서·게이트웨이 Heartbeat를 모두 고려해야 합니다.

Inactivity 시간 공식:

Inactivity Duration = (센서 HB × 2) + (게이트웨이 HB + 1분)

예: 센서 Heartbeat 60분, 게이트웨이 Heartbeat 5분 → 최소 126분 = (60 × 2) + (5 + 1).

Inactivity 알림 문제 해결: (1) 알림을 트리거한 장치 확인(알림에 센서 이름 표시) → (2) imonnit.com 로그인해 오프라인 확인 → (3) 오프라인이 아니면 Inactivity 시간 설정 확인.

  • 특정 장치만 오프라인: Overview에서 마지막 체크인 시각·신호 강도(약하면 원인)·배터리(낮으면 원인) 확인, 리피터 사용 시 리피터 온라인·메시지 처리 확인, 미사용 시 게이트웨이 확인.
  • 모든 센서 오프라인: 게이트웨이부터 확인(켜짐·체크인 여부; 셀룰러는 셀 네트워크, Ethernet은 인터넷 접근, USB는 유틸리티 상태 "Active"). https://status.monnit.com/ 에서 서버 문제 확인 가능.

iMonnit 규칙의 Snooze 설정 이해

Rules

Snooze는 규칙이 Disarm되지 않으면 구성된 시간 간격으로 알림을 계속 보내는 설정입니다(알람 시계 스누즈와 유사). Disarm(Acknowledge)될 때까지 알림이 계속됩니다.

기본 60분 구성: 규칙이 트리거되고 60분 Snooze 시간 내에 Disarm되지 않으면, 조건이 지속되는 한 60분마다 반복 알림이 발송됩니다. Disarm되고 Rearm되지 않으면 조건이 지속되어도 Snooze 간격 알림이 더 이상 발송되지 않습니다.

Snooze 시간 구성: 메뉴 Rules → 규칙 선택 → 상단 Edit 아이콘(연필) → "Snooze: Don't Alert again for (Minutes):" 필드에 분 입력 → Save.

예시 (ALTA 온도 센서, 45°F 초과 시 트리거, 기본 60분 Snooze, 10분 Heartbeat): 센서가 46°F로 트리거되면 이메일 발송·Snooze 타이머 시작 → 조건 지속 중 60분 도달 시마다 추가 이메일 발송·타이머 재시작 → 사용자가 Disarm하면 Snooze 타이머 중지(조건 지속되어도 미발송) → 센서가 임계값 미만(44°F) 보고 시 Rearm(Reset)되어 다시 트리거 가능.

규칙 - System Action 이해

Rules

System Action은 알림 및/또는 Add-on 장치(Control Unit, Local Alert 등)와 관련된 로직을 구성합니다. Tasks 탭과 함께 사용됩니다. 규칙이 트리거되면 트리거 규칙에 구성된 System Action이 대상 규칙(동일하거나 완전히 다른 규칙)에 실행됩니다. (Monnit Support는 System Action 로직 설계·검토 서비스를 제공하지 않고 문서로 안내합니다.)

4가지 System Action: (1) Acknowledge(Disarm), (2) Full Reset(Rearm), (3) Activate, (4) Deactivate.

  • Acknowledge 사용 사례: 규칙이 트리거 후 자신을 Disarm하여, 트리거 조건에 단일 알림(예: SMS)만 보내고 Snooze 타이머(기본 60분) 반복 알림을 막음. 조건 정상 복귀 시 Auto-Acknowledge가 활성화되어 있으면 Rearm됨.
  • Full Reset 사용 사례: 규칙이 자신에게 Full Reset을 실행하면 센서가 트리거 조건을 보고할 때마다 Disarm/Rearm되어 매번 알림 트리거(Snooze 대신).
  • Activate/Deactivate: 대상 규칙을 켜거나 끔(트리거 완전 활성화/비활성화).

Delay 추가 고려: 각 System Action에 Delay 설정 가능(계획된 순서 로직에 유용). Delay 없으면 즉시 실행되므로, 다른 작업과 순서가 어긋나지 않도록 1분 지연 권장.

System Action 생성: 규칙 생성/선택 → Tasks 탭 → Create System Action → Action 선택 → Delay 선택 → Target Rule 선택 → Add.

예약 규칙과 함께 사용: Scheduled Rule(반복 월/주/일/시/15분 일정)을 생성하면 원하는 시간에 System Action을 실행할 수 있어 강력합니다.

System Action 이력: 규칙 선택 → History 탭 → 날짜 범위 설정 → Set → 실행 순서대로 System Action 확인.

Sensor Reading 규칙과 Advanced 규칙의 차이

Rules

흔한 오해: 많은 신규 고객이 센서의 Aware State Threshold를 설정하면 알림이 트리거될 것으로 생각하지만, 센서의 Aware State 동작은 사용자가 해당 구성을 하지 않는 한 규칙과 연결되지 않습니다. Aware State 구성은 센서 통신·동작을 제어하지만, 대응 규칙을 생성하고 센서를 할당하지 않으면 어떤 규칙과도 연결되지 않습니다.

  • Sensor Reading 규칙: Condition 구성이 규칙 Condition 탭에 있으며, 센서의 Aware State Threshold와 본질적으로 연결되지 않습니다. 센서 판독값과 규칙에 설정한 트리거 포인트(Greater Than, Less Than, Equal to 등)로만 트리거됩니다.
  • Advanced 규칙: 센서의 Aware State(및 기타 기록 조건)로 트리거되며, Aware State Threshold 설정과 직접 연결됩니다. 더 복잡한 로직 가능(예: "x"회 연속 Aware 메시지). 가장 일반적인 것은 Notify After Aware Period. Advanced 규칙은 Premiere 구독자만 사용 가능.

센서의 Aware State Threshold가 규칙을 트리거하지 않는다면 규칙 유형을 확인하세요(Aware State 기반 트리거는 Advanced 규칙만 가능).

규칙 또는 센서 일정의 "Before and After" 이해

Rules

iMonnit 소프트웨어는 하루를 오전 12:00:00부터 오후 11:59:59까지 24시간 단위로 정의합니다.

  • "Between": 단일 하루 내에서 연속 "on" 기간을 설정. 일정이 하루의 끝(11:59:59 PM)과 새 날의 시작(12:00:00 AM) 경계를 넘으면 적용되지 않습니다(시작 시간이 종료 시간보다 늦으면 적용 안 됨).
  • "Before and After": "Between"의 반대로, 단일 하루의 연속 기간을 "off" 시간으로 설정하여 하루에서 다음 날로의 전환 동안 일정이 활성화되게 합니다. Before 시간을 PM, After 시간을 AM으로 설정하면 일정이 무효화됩니다.

예시 (낮에는 끄고 밤새 다음 날 아침까지 활성): 비활성 8 AM~6 PM, 활성 6 PM~다음 날 8 AM → Before 8 AM and After 6 PM 선택. (예: 동작 감지 센서를 영업시간에는 끄고 영업 외 시간에 동작 감지 시 알림.)

규칙이 구성된 일정 외에 계속 알림을 보내는 경우

Rules

규칙에 일정을 적용하면 지정 기간에만 알림을 받습니다. 그러나 규칙이 활성 일정 중 트리거되고 비활성 예약 전에 Disarm되지 않으면 Snooze 시간(기본 60분)에 계속 발송됩니다(예약 외 시간에도). 이때 Snooze 알림은 일정 내 트리거한 마지막 판독값을 표시합니다.

해결책 (System Action으로 자동 Rearm): 비활성 예약 시간에 원래 규칙을 자동 Rearm하는 별도 Scheduled Rule 생성:

  1. 새 Scheduled 규칙 생성.
  2. 원래 규칙의 예약 off 시간과 동일하게 실행 시간 설정.
  3. Task 구성에서 System Action 버튼 선택(활성화).
  4. "Action to be done"을 Full Reset으로 설정.
  5. Delay를 No Delay로.
  6. "Target Notification"을 비활성화할 규칙으로 설정.
  7. Save.

이 규칙으로 원래 규칙이 비활성 예약 시간에 자동 Rearm되어 Snooze 시간에 더 이상 발송하지 않습니다. (Scheduled Rule 자체도 Disarm/Rearm이 필요하므로 자신에게 Full Reset System Action을 추가하여 다음 날 실행되게 함.)

로깅된 데이터: 센서/게이트웨이가 데이터를 로깅하고 규칙 활성 일정 중 업로드되면, 비활성 시간에 기록된 판독값이어도 규칙을 트리거합니다.

규칙 - Notify After Aware Period

Rules

서버가 센서 메시지를 받으면 규칙·트리거를 확인합니다. 활성 "Notify After Aware Period" 알림이 있으면 현재 데이터가 Aware State인지 계산하고, 이전 메시지와 비교해 센서가 Aware State였던 시간을 판단합니다. "Alert After" 설정 시간보다 길게 Aware였으면 알림을 보냅니다.

Aware 메시지는 게이트웨이에서 통신을 트리거하지만, 표준(비-Aware) 메시지는 그렇지 않습니다. 서버는 Aware 메시지 전달을 수 초 내에 알지만, 센서가 정상으로 복귀하면 비-Aware 메시지는 정규 Heartbeat까지 게이트웨이에 큐잉되어 즉시 전달되지 않습니다. 따라서 Notify After 타이머를 Aware State Heartbeat보다 약간 짧게 설정하는 것이 권장됩니다(두 번째 메시지가 올 때 타이머가 이미 경과하도록).

예시:

  • Aware Heartbeat 30분, Notify After 29분: 1:00 첫 Aware → 1:30 두 번째 Aware(30분 > 29분이므로 알림 발송).
  • Aware Heartbeat 30분, Notify After 31분: 1:30 두 번째(30분 < 31분, 미발송) → 2:00(60분 > 31분, 발송).
  • Aware Heartbeat 15분, Notify After 29분: 1:15(15분 < 29분, 미발송) → 1:30(30분 > 29분, 발송).

센서 Heartbeat, Aware State, Assessment, Rearm 이해

Sensors

Monnit 센서 작동은 Aware State 개념에 기반합니다. 조건이 감지되는 즉시 보고할 수 있게 하는 센서 작동 상태입니다.

Event 센서 vs Interval 센서:

  • Event 센서: 이벤트 발생 즉시 반응(yes/no 데이터). 예: Water Detection, Door/Window, Motion, Dry Contact.
  • Interval(Reading) 센서: 시간 기반으로 설정 간격마다 깨어나 데이터 수집·전송 후 수면. 예: Temperature, Humidity, Voltage, Pressure.

Heartbeat: 무료 Basic은 최소 120분, 유료 Premiere는 최소 10분(Aware State Heartbeat·Assessment 등 고급 설정 포함). 센서는 배터리 절약을 위해 스트리밍 대신 Heartbeat 간격으로 전송(조립 시 기본 120분, 구성 가능). Event 센서는 Heartbeat로 활성 상태·현재 상태·신호·배터리 보고, Reading 센서는 정보 수집·전송 간격으로 사용.

Aware State: 사전 구성된 조건 감지 시 즉시 iMonnit으로 전송. 즉시 전송하려면 게이트웨이의 On Aware Messages(Force Transmit on Aware) 설정이 Trigger Heartbeat(On)여야 함(기본 활성화). Event 센서는 이벤트 감지가 자동으로 Aware State 트리거, Reading 센서는 사용자가 조건 설정. Aware State Heartbeat를 빠른 간격으로 설정하면 정상 복귀까지 더 자주 정보·알림 수신. Aware 플래그 메시지는 게이트웨이가 즉시 서버로 통신(표준 메시지는 다음 게이트웨이 Heartbeat까지 큐잉).

Assessments per Heartbeat: Reading 센서가 깨어나 데이터를 수집하고 Aware State 조건과 비교. 충족 시 즉시 전송(아니면 수면). 자주 Assessment할수록 Aware 조건을 빠르게 감지. 트리거 조건을 충족하지 않는 Assessment는 폐기됨(데이터 포인트로 변환 안 됨, Aware State Heartbeat 트리거 시 제외).

Time to Rearm: Event 센서가 다음 이벤트를 감지하기까지의 시간 정의(예: 동작 센서 Rearm 1초 = 1초 후 재감지 가능, 더 자주 정보 수집하나 배터리 소모↑; 10분 = 배터리 절약).

Aware State 작동 예시 (Heartbeat 120분, Aware Heartbeat 30분, Assessments 2회/60분, 임계값 257°F):

  • 정상 시: 60분마다 Assessment, 120분마다 Heartbeat 전송.
  • 4:00 PM 258°F(임계값 초과) → Aware State 진입, urgent 태그로 즉시 전송, Heartbeat 30분·Assessment 15분으로 변경.
  • Aware 중 Assessment(Heartbeat 아님)는 전송 안 함, Heartbeat 시각에만 전송.
  • 5:15 PM 256°F(임계값 미만) → Aware State 이탈 → 5:30 PM 전송, Heartbeat 120분·Assessment 60분으로 복귀.

iMonnit의 센서 설정(Sensor Settings)

Sensors

  • Assessments Per Heartbeat: Heartbeat 사이에 임계값 대비 측정값을 확인하는 횟수.
  • Aware State Threshold: Aware State 진입 전 기록할 최소/최대 판독값.
  • Aware State Buffer: 임계값에 매우 근접할 때 표준 작동과 Aware State 사이를 오가는 것을 방지(예: 최대 임계값 90도, 버퍼 1 → 90.1도에서 Aware, 89.0도로 떨어져야 정상 복귀).
  • Sync: 센서 10개 이하 소규모 네트워크에서 통신 동기화(기본 off는 통신 무작위화로 견고성 극대화).
  • Failed Transmissions Before Link Mode: 게이트웨이 응답 없이 배터리 절약 link 모드로 전환하기 전 전송 횟수(낮으면 더 적은 누락으로 새 게이트웨이 찾음, 높으면 노이즈 환경에서 현재 게이트웨이 유지, 0이면 절대 다른 게이트웨이에 합류 안 함 — 배터리 분리·재삽입 필요).

설정 변경 후 Save 클릭. 설정은 즉시 적용되지 않고 다음 Heartbeat에 적용됩니다(대기 중 "Fields waiting to be written to the sensor..." 메시지). 기본 Heartbeat 120분이며, 잦은 Heartbeat·Assessment는 배터리 수명에 큰 영향을 줍니다.

iMonnit Heartbeat 설명

Sensors

[동영상: https://www.youtube.com/embed/xsfKE297LOM]

Heartbeat란? 예외(Aware State) 활동이 감지되지 않을 때 센서가 게이트웨이와 판독값을 통신하는 빈도(분)입니다. 기본 120분(2시간), Premiere로 최소 10분, 로컬 호스팅(Express/Enterprise/MINE)은 더 잦게 구성 가능.

Assessments Per Heartbeat: 센서 Heartbeat·Aware State Heartbeat와 함께 작동. 센서는 예약 Heartbeat에만 전송하지만 사이에 Assessment(샘플 판독)로 Aware State Threshold 위반을 확인. 위반 시 깨어나 게이트웨이로 전송하고 Aware State Heartbeat 빈도로 체크인 시작.

게이트웨이 Heartbeat: 게이트웨이도 Heartbeat가 있으며 유형별로 기본값이 다름. 센서는 센서 Heartbeat에 게이트웨이로 전송하고, 게이트웨이는 게이트웨이 Heartbeat에 iMonnit으로 전송(지연 발생 가능). 게이트웨이 Heartbeat를 가장 잦은 센서 Heartbeat보다 잦게 설정하는 것이 좋습니다.

게이트웨이별 기본 Heartbeat: Ethernet 5분, 4G LTE 15분, 3G 셀룰러 60분, 2G 셀룰러 60분, Wireless Sensor Adapter 5분, Serial MODBUS는 Heartbeat 없음.

Force Transmit on Aware: 게이트웨이 기본 설정으로, 센서가 Aware State에 진입하면 예약 Heartbeat 전에 즉시 iMonnit으로 전송(예외 상태 즉시 반영의 핵심 기능).

편집: Sensors → 센서 선택 → Settings 탭(기어 아이콘) → Heartbeat Interval·Aware State Heartbeat·Assessments per Heartbeat 입력 → Save.

(작동 예시는 "센서 Heartbeat, Aware State, Assessment, Rearm 이해" 문서와 동일.)

Authenticator 앱으로 2FA 설정

Users

2FA(2단계 인증)는 비밀번호 외 동적 인증 코드를 요구하여 무단 접근 위험을 크게 줄입니다. Google Authenticator 또는 Microsoft Authenticator로 활성화합니다.

사전 요건: 활성 계정, 위 앱 중 하나가 설치된 스마트폰.

단계:

  1. 2FA 설정 이동: iMonnit 로그인 → 왼쪽 메뉴 Users → 사용자 선택 → 우상단 2FA 탭(열쇠 아이콘).
  2. Authenticator App 옵션 선택: Authenticator App 섹션의 아래 화살표(v)로 펼침.
  3. 설정 코드 생성: Generate New Code → QR Code와 Manual Entry Key 표시. (모바일폰 사용 시 QR 스캔 불가하므로 Manual Entry Key 복사.)
  4. Authenticator 앱 구성: Google은 "+" → Scan a QR code, Microsoft는 "+"/Add Account → Other → QR 스캔. (카메라 스캔 불가 시 앱에서 수동 입력으로 secret key 입력.)
  5. 확인·연결: 앱에 "iMonnit" 6자리 코드 표시 → iMonnit 페이지 인증 필드에 입력 → Verify/Enable.

Remembered Devices 관리: 2FA 활성화 후 2FA 대시보드 오른쪽에서 확인. Re-Auth In은 해당 기기가 새 2FA 코드를 요구하기까지 남은 일수, Revoke Access(휴지통 아이콘)는 다음 로그인 시 2FA 강제.

보안 팁: 휴대폰 분실 시 접근 손실을 막기 위해 기기를 백업하거나 복구 코드를 기록하세요.

iMonnit HX 작동 방식

iMonnit HX

iMonnit Online의 표준 최소 Heartbeat는 10분입니다. 더 잦은 Heartbeat가 필요한 고객을 위해 Monnit은 iMonnit HX Credit을 제공하여 센서 Heartbeat를 최소 1분까지 설정할 수 있게 합니다. Premiere 사용자는 계정에 충분한 크레딧이 있는 한 1분 Heartbeat를 적용하고 HX Credit으로 데이터 포인트를 표시합니다.

운영 세부사항: iMonnit HX Activation Key를 적용하면 계정이 HX Enabled되어 1분 Heartbeat 구성 가능. 센서가 하루에 10분보다 자주 체크인하지 않으면 크레딧이 소모되지 않습니다. 일일 한도 150 레코드를 초과하는 모든 Heartbeat가 HX Credit 1개씩 사용:

  • 1 센서 × 10분 Heartbeat = 시간당 6회 = 하루 144회
  • + Aware 판독용 6회 = 하루 150회

하루 150 데이터 포인트 초과 시 초과분마다 HX Credit 1개 사용.

요건: 활성 Premiere 계정, 1분 Heartbeat 지원 센서, 충분한 HX Credit.

HX Credit 소진 시: 센서가 계속 잦은 Heartbeat로 체크인하면 150 임계값 초과 데이터가 숨겨짐(Readings 탭에 "HX Message Credits Consumed" 표시). 추가 구매하거나 support@monnit.com에 HX 비활성화 요청, 그리고 Heartbeat를 10분 이상으로 설정해야 함.

  • Webhook: HX Credit이 완전히 소진되면 데이터 접근이 제한되고 Webhook이 비활성화됨.
  • 중요: 추가 크레딧 구매 시 소진 이후 발생한 Heartbeat 부족분에 먼저 적용되므로, 구매 후 잔액은 부족분 처리 후 남은 양입니다.
  • 낮은 HX Credit 경고 알림 임계값 구성 가능(잔액이 임계값 미만이면 알림 발송).

게이트웨이 대역폭 제한: 1분 Heartbeat 운영 시 게이트웨이 대역폭 고려 필요. 대부분 게이트웨이는 최대 100개 센서 지원하나, 빠른 Heartbeat는 센서당 무선 대역폭을 줄입니다. 모범 사례: 게이트웨이당 100개 센서 지원에 최소 2분 Heartbeat 권장. 1분 Heartbeat가 필요하면 게이트웨이당 최대 50개 센서로 제한.

구매·적용: Monnit.com 온라인 스토어에서 iMonnit HX Credits 구매 → Quantity 선택·Checkout → Wallet 탭에서 결제 → 구매 후 이메일로 Activation Key 수신 → HX Credits 코드 입력란에 입력·적용.
